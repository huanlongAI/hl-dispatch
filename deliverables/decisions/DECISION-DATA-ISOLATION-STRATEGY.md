**文档编号**：DECISION-2026-03-04-DATA-ISOLATION
**版本**：v2.0（2026-03-04 裁决：X-C 混合方案取代原方案 C 全量迁移）
**裁决人**：童正辉（创始人）
**接收人**：曾正龙（运维）、许久明（架构）、魏鹏（后端）、刘建成（PM）
**状态**：🔒 已裁决，待团队确认执行细节

---

## 一、裁决结论

**Phase 1 采用 X-C 混合方案：治理必需子集迁移 + 能力包保持 MySQL。**

核心原则：**旧系统数据库完全不动，HK 内核在 PG18 中只建治理专属表，两者通过 API 桥接。**

本裁决替代原 INQUIRY-DATA-SYNC-STRATEGY v1.1 中的方案 C（一次性全量迁移 + 停写切换）。

---

## 二、裁决背景

### 曾正龙提出的两个关键问题

| # | 问题 | 风险 |
|---|------|------|
| 1 | 单次同步需要停机并全量更新服务，风险不可控 | 维护窗口内所有依赖基础数据的服务同时切换，big-bang 风险 |
| 2 | 数据库改造期间是否需要代码冻结？不冻结就会出现数据清洗永远追不上开发进度 | Schema 映射 + ETL 脚本反复重写，WS-C 开发被阻塞 |

### 创始人的前提质疑

> 旧系统作为能力包，不能保持 MySQL 数据库吗？为什么必须强行统一到 PG？

经分析：R-031 仅要求迁移「基础数据」（身份/租户/组织），业务数据从未要求迁移。且 HK 内核的治理决策大部分依赖自有数据（策略、审计、灰度配置），只有灰度路由和能力包绑定需要少量租户映射数据。

**结论：全量迁移基础数据是过度工程化，应最小化迁移范围。**

## 三、X-C 方案详述

### 3.1 数据四区划分

| 分区 | 存储位置 | 数据内容 | 写入方 | 数据量 |
|------|---------|---------|--------|--------|
| **治理区**（HK 内核自有） | PG18 | 策略定义、审计日志、灰度配置、reason_code 字典、feature-toggles | HK 内核独占写入 | HK 内核新建，无迁移 |
| **映射区**（治理必需子集） | PG18 | 租户→灰度分组、租户→能力包绑定 | 管理 API（运维/创始人操作） | 百~千行级，无迁移，新建录入 |
| **业务区**（能力包自有） | 旧 MySQL 8 | 订单、库存、商品、支付、客户资产等 | 旧系统写入，**不迁移** | 原地不动 |
| **基础数据区** | 旧 MySQL 8 | 身份、租户详情、组织架构树 | 旧系统写入，HK 通过 API 读取 | **原地不动，Phase 2 退役时再迁移** |

### 3.2 HK 内核各模块数据来源

| HK 模块 | 需要什么数据 | X-C 方案下数据来源 |
|---------|------------|------------------|
| GrayscaleRoutingFilter | 租户灰度分组 | PG18 映射区 `hk_tenant_grayscale`（本地查询，高频热路径） |
| FeatureToggleFilter | 模块启用状态 | Nacos 配置 `feature-toggles.yaml`（不需要数据库） |
| Can*Service | 策略规则 | PG18 治理区 `hk_policy_*`（HK 自有数据） |
| AuditService | 审计日志 | PG18 治理区 `hk_audit_log`（HK 自有数据） |
| Gateway 后置检查 | 租户能力包绑定 | PG18 映射区 `hk_tenant_capability`（本地查询） |
| 控制台展示 | 租户详情、组织树 | 旧 MySQL API（通过适配层读取，非治理热路径） |

### 3.3 映射区表定义（草案）

仅需 **2 张配置表**，数据由管理 API 写入，非迁移：

```sql
-- 表1：租户灰度分组（GrayscaleRoutingFilter 查询）
CREATE TABLE hk_tenant_grayscale (
    tenant_id   BIGINT PRIMARY KEY,           -- 旧系统租户ID
    group_name  VARCHAR(50) NOT NULL,          -- 灰度分组（canary / stable）
    percentage  INT NOT NULL DEFAULT 0,        -- 分组流量百分比
    updated_at  TIMESTAMPTZ DEFAULT NOW()
);

-- 表2：租户能力包绑定（Gateway 权限校验）
CREATE TABLE hk_tenant_capability (
    tenant_id       BIGINT NOT NULL,
    capability_key  VARCHAR(100) NOT NULL,     -- 如 biz.customer.asset
    enabled         BOOLEAN DEFAULT TRUE,
    PRIMARY KEY (tenant_id, capability_key)
);
```

## 四、对曾正龙两个问题的回应

| 原问题 | X-C 下的状态 |
|--------|-------------|
| Q1：停机 + 全量更新风险 | **消失**。不存在全量 ETL。映射区是全新配置表，在 PG18 直接 CREATE TABLE + INSERT 即可，耗时分钟级，无需停机 |
| Q2：代码冻结 / 数据清洗追赶 | **消失**。旧系统 MySQL 完全不动，不存在 Schema 映射和 ETL 脚本，WS-C 开发零阻塞 |

## 五、R-031 修订

| R-031 原条件 | X-C 下的变化 | 新状态 |
|-------------|-------------|--------|
| 条件 1：仅限基础数据迁移 | → Phase 1 不迁移基础数据，留旧 MySQL，API 读取 | **修订** |
| 条件 2：ID 映射表 | → 映射区直接使用旧系统原 tenant_id，无需 ID 映射 | **简化** |
| 条件 3：对账作业 | → 不存在双写，无需对账 | **取消** |
| 条件 4：统一写入入口 | → 各区独立写入方，职责清晰 | **修订** |
| 条件 5：Phase 2 退役 | → Phase 2 退役旧系统时做离线迁移（旧系统已下线，更安全） | **保留，延后执行** |

**R-031 实质上从"数据同步规范"变为"数据隔离规范"**。

## 六、R-034 Sprint 影响

| Sprint | 原交付物 | X-C 下的变化 |
|--------|---------|-------------|
| S1 | Schema 映射 + ETL 脚本 + 试迁演练 | → **移除**。替换为：映射区 2 张表 DDL + 管理 API 实现 + 适配层 API（读旧系统）评估 |
| S2 | — | → 不变（HK 7 模块投产 + Gateway，映射区表已就绪） |
| S3 | 能力包端到端验证 + Canal 对账 | → **移除 Canal 对账**。能力包端到端验证照常，数据从旧 MySQL 直接读取 |
| S5 | 灰度切流 | → 不变（灰度配置通过 `hk_tenant_grayscale` 管理） |

**S1 工作量大幅降低**：从"ETL 脚本开发 + Schema 映射 + 试迁演练"（预计 2~3 周）降为"2 张表 DDL + 管理 API"（预计 2~3 天）。

## 八、风险

| 风险 | 严重度 | 缓解 |
|------|--------|------|
| 旧 MySQL 成为治理链路依赖（控制台展示需要 API 调用） | 低 | 控制台展示非治理热路径，旧系统在 Phase 1 仍在运行 |
| 映射区配置数据需手动维护 | 低 | 数据量极小（百~千行），S4 灰度操作 API 完成后可自动化 |
| Phase 2 退役时仍需迁移基础数据 | 中 | 届时旧系统已下线，是纯离线迁移，无停写窗口、无冻结问题，比 Phase 1 在线迁移简单得多 |
