## 一、问题陈述

R-031 原定使用 Canal 作为 MySQL→PG 的基础数据过渡同步工具，但经团队评估发现：

1. **Canal 原生不支持 PG 作为目标端**：Canal 架构为 MySQL binlog → Canal Server → 下游，目标端仅支持 MySQL/RDS MySQL
2. **Canal 稳定性存疑**：基于 JDK 1.8，2024 年 1.0→2.0 迁移中表现不可用
3. **替代方案（DTS）未验证**：阿里云 DTS 是否支持 PG18 作为目标端未经测试，且可能存在表结构差异需二次开发

R-031 的五项限定条件本身仍然有效（仅基础数据、ID 映射、对账、统一入口、Phase 2 退役），但**实现工具需要重新选型**。

## 二、方案 C 重点分析 — 一次性全量迁移 + 停写切换

### 2.1 方案描述

选定一个维护窗口，将旧 MySQL 8 中 R-031 范围内的基础数据（身份/租户/组织等）一次性全量导入 PG18，切换完成后新系统成为这些数据的唯一写入端。

```
切换前：旧 MySQL 8（读+写）→ 业务正常运行
         ↓
维护窗口：旧 MySQL 8（停写）→ 全量 ETL → PG18（验证）→ 切换读写端点
         ↓
切换后：PG18（读+写）→ 旧 MySQL 8（只读归档）
```

### 2.2 为什么方案 C 可行性高

R-031 限定了同步范围为**基础数据**（身份/租户/组织），这类数据具备以下特征：

| 特征 | 对方案 C 的影响 |
|------|---------------|
| 写入频率低 | 维护窗口内增量极少，全量导入即可覆盖 |
| 数据量可控 | 租户/组织/身份表通常在万~十万级，ETL 耗时短 |
| 强审计要求 | 一次性迁移后单一写入端比双写更容易保证审计完整性 |
| Phase 2 退役要求 | 无需建设后续要拆除的同步基础设施，直接一步到位 |

### 2.3 DDL 迁移规范要点 `[v1.1 补丁]`

Schema 映射不是简单的类型替换，需遵循以下规范（魏鹏在步骤 2 中负责落实）：

| MySQL 特性 | PG18 目标 | 注意事项 |
|-----------|----------|---------|
| `AUTO_INCREMENT` | `GENERATED ALWAYS AS IDENTITY` | **不用 SERIAL**（PG18 推荐 IDENTITY，Sequence 所有权更清晰） |
| `DATETIME` | `TIMESTAMPTZ` | 必须统一为 UTC 存储，应用层转换时区 |
| `ENUM('a','b','c')` | `CHECK` 约束或独立字典表 | 纯 CHECK 会丧失集中枚举治理能力；如果枚举需要跨表引用，用字典表 |
| `TINYINT(1)` | `BOOLEAN` | — |
| `VARCHAR(N)` charset | `VARCHAR(N)`（PG 默认 UTF-8） | 确认 MySQL 源表 charset 为 utf8mb4，否则迁移可能截断 |
| `JSON` | `JSONB` | PG 的 JSONB 支持索引和查询，优于 JSON |
| MySQL B-tree 索引 | PG B-tree（默认） | 索引名可能冲突（PG 索引全局命名空间），需加表名前缀 |
| `DEFAULT CURRENT_TIMESTAMP` | `DEFAULT NOW()` | 语法差异，自动处理 |

### 2.4 方案 C 的执行流程（草案） `[原§2.3，重编号]`

| 步骤 | 操作 | 负责人 | 预估耗时 |
|------|------|--------|---------|
| 1. 数据盘点 | 确认 R-031 范围内的源表清单、行数、总大小 | 魏鹏 + Java 组 | 1~2 天 |
| 2. Schema 映射 | MySQL DDL → PG DDL 转换，标注差异（自增→SEQUENCE、ENUM→CHECK、字符集等） | 魏鹏 | 2~3 天 |
| 3. ETL 脚本开发 | 选择工具（pgloader / 自研 Python 脚本 / DBeaver 导出导入）+ ID 映射表生成 | 魏鹏 + 曾正龙 | 3~5 天 |
| 4. 试迁演练 | 在 Staging 环境执行完整迁移流程，记录耗时和问题 | 曾正龙 | 1~2 天 |
| 5. 对账验证脚本 `[v1.1]` | 三层对账：L1 行数级 + L2 约束级（唯一键/外键/值域）+ L3 回归级（关键查询用例）+ ID 映射完整性 | 测试组 + 魏鹏 | 2~3 天 |
| 6. 正式迁移 `[v1.1 细化]` | 见下方§2.4 停写技术方案，含数据库层+应用层双重停写 | 曾正龙 + 魏鹏 | 维护窗口 2~4 小时 |
| 7. 旧库只读归档 | MySQL 8 设为 `read_only=ON`（全局），保留 90 天供回查（**非回切**，见§2.6） | 曾正龙 | 切换后立即 |

### 2.4 停写技术方案 `[v1.1 补丁 D1]`

"停写"不是口头约定，必须有技术门禁保障。正式迁移窗口内的操作序列：

| 步骤 | 操作 | 验证 |
|------|------|------|
| 1 | **应用层维护态开关**：Nacos 配置 `maintenance.mode=true`，所有基础数据写入接口返回 `503 Service Unavailable`（body 含 `reason_code=MAINTENANCE_WINDOW`） | curl 写入接口返回 503 |
| 2 | **数据库层只读**：`SET GLOBAL read_only=ON;`（MySQL 8，super 用户除外）| `SHOW VARIABLES LIKE 'read_only';` = ON |
| 3 | **等待 in-flight 事务完成**：`SELECT * FROM information_schema.INNODB_TRX;` 确认无活跃写事务 | 活跃事务数 = 0 |
| 4 | **执行全量 ETL** | pgloader / 脚本日志无 ERROR |
| 5 | **执行对账验证**（§2.5） | 三层对账全部通过 |
| 6 | **切换连接端点**：应用配置指向 PG18 | 连接池健康检查通过 |
| 7 | **关闭维护态**：`maintenance.mode=false` | 写入接口恢复 200 |

**关键约束**：步骤 1~2 必须在 ETL 开始前完成，确保源库在导出期间无新写入。步骤 3 是安全等待，超过 60s 仍有活跃事务 → 终止迁移、恢复写入。

### 2.5 对账策略升级 `[v1.1 补丁 D2]`

原方案仅含"行数比对+抽样"，不足以覆盖关键不变量。升级为三层对账：

| 层级 | 检查内容 | 通过标准 | 工具/方法 |
|------|---------|---------|----------|
| L1 行数级 | 每张表行数 MySQL = PG | 100% 一致 | SQL COUNT(*) 比对脚本 |
| L2 约束级 `[v1.1]` | 唯一键无重复 + 外键引用完整 + 枚举值域合法 + 非空约束无违反 + 时间戳范围合理 | 100% 通过 | PG 端 `pg_catalog` 约束验证 + 自定义校验 SQL |
| L3 回归级 `[v1.1]` | 关键业务查询回归用例（如：按 tenant_id 查组织树、按 user_id 查身份信息、按 org_id 查策略绑定）| 全部返回预期结果 | 预定义查询用例 + 断言脚本 |

**对账不通过的处理**：任一层级不通过 → 中止切换 → 恢复 MySQL 写入（关闭 read_only + 关闭维护态）→ 分析差异原因 → 修复后重新执行。

### 2.6 回退定义 `[v1.1 补丁 D3]`

**明确声明：回退 = 回读旧库止血，不承诺回切写入。**

| 场景 | 回退动作 | 限制 |
|------|---------|------|
| 切换后 24h 内发现数据问题 | 将读请求指回旧 MySQL（只读），新系统暂停写入 | 旧库数据冻结在切换时刻，不含切换后的新写入 |
| 切换后 24h+ | 旧库仅作回查参考，不再作为回退目标 | 需要针对 PG 端问题直接修复 |
| 需要"回切写入" | 必须建设反向迁移（PG→MySQL），成本极高 | **Phase 1 不建设反向迁移**，如需此能力需额外立项 |

**旧库保留策略**：`read_only=ON` 保留 90 天，到期前由曾正龙 + 魏鹏评估是否可下线。90 天内只允许读取（回查/审计），禁止任何写入。

### 2.7 方案 C 的风险与缓解 `[原§2.4，重编号]`

| 风险 | 严重度 | 缓解措施 |
|------|--------|---------|
| 维护窗口期间业务不可用 | 中 | 选择凌晨低峰期；基础数据写入频率低，窗口预计 2~4 小时可完成 |
| MySQL↔PG 数据类型不兼容 | 中 | Schema 映射阶段逐一排查；PG18 推荐 `GENERATED ALWAYS AS IDENTITY`（非 SERIAL）；`DATETIME`→`TIMESTAMPTZ`；`TINYINT(1)`→`BOOLEAN`；`ENUM`→`CHECK` 或独立字典表（保留集中治理能力）`[v1.1]` |
| 迁移后发现数据错误 | 高 | 三层对账（§2.5）必须在切换前全部通过；旧库保留只读 90 天作为**回查**依据（非回切，见§2.6）`[v1.1]` |
| 旧系统仍有模块读旧库 | 中 | 迁移前盘点所有连接源（见 Q12 连接源盘点方法）`[v1.1]`，确保全部切到 PG18 |
| 能力包适配层依赖旧 ID | 中 | ID 映射表持久化（R-031 条件 2），适配层查询时先查映射 |

### 2.8 方案 C 的优势总结 `[原§2.5，重编号]`

相比建设 Canal/DTS 增量同步链路，方案 C 具备以下优势：

- **零同步基础设施**：不引入 Canal/DTS/CDC 组件，减少运维复杂度和故障面
- **数据一致性强**：切换后只有一个写入端，不存在双写冲突和延迟窗口
- **R-031 条件 5 自动满足**：无需 Phase 2 退役同步链路，因为根本没有
- **成本低**：不产生 DTS 按量费用，不需要维护同步监控
- **可回查**：旧库保留只读 90 天，可供回查审计；回退定义见§2.6（回读止血，非回切写入）`[v1.1 修正]`

| 指标 | 阈值 | 说明 |

| 演练全流程耗时 | ≤ 维护窗口上限的 **50%** | 留 50% 缓冲应对意外（如网络抖动、数据量偏差） |

| L2 约束检查 | **100%** 通过 | 唯一键 / 外键 / 值域 / 非空 / 时间范围全部合规 |
| L3 回归查询 | **100%** 通过 | 预定义的关键业务查询返回预期结果 |
| ID 映射完整性 | **100%** | 映射表覆盖所有迁移记录，无孤儿 ID |
| 连接源盘点签字 | **全员确认** | Q12 盘点结果由各模块负责人签字确认 |

**演练不通过的处理**：分析失败原因 → 修复 → 重新演练。连续 2 次不通过 → 回退到备选方案评估。
