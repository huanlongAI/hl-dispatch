## 1. 验收原则

- P0 验收只验证“AI 老板总助 App P0 文档和后续实现是否满足边界”，不声称产品已实现。
- 任一验收项不得依赖真实门店经营系统数据。
- 任一验收项不得依赖智能眼镜、录音卡、持续录音或后台录音。
- 任一验收项不得验证自主执行能力。
- 中国区生产态必须使用合规白名单模型，审核态等于真实生产态。

## 6. 24h 观察协议

观察目标：确认 P0 核心链路是否稳定、合规边界是否被明显突破。

必看指标：
- 新用户完成首次提问率。
- 文字输入成功率。
- 语音转写成功率。
- AI 生成成功率。
- 合规拒答命中样本。
- 崩溃、卡死、明显 UI 错位。
- 用户主动反馈中的高频困惑。

阻断条件：
- 生产调用链出现非白名单模型。
- 审核态与生产态不一致。
- 输出出现医疗越界。
- App 暗示已执行真实系统动作。

## 7. 72h 观察协议

观察目标：确认模板质量、用量策略、老板真实使用场景是否成立。

必看指标：
- 人均对话轮次。
- 模板使用占比。
- 复制/分享/收藏比例。
- 重新生成比例。
- 用量不足触发次数。
- 合规拒答后的继续提问率。
- 用户反馈中“无法落地”“太泛”“不敢用”的比例。

处理策略：
- 对高频失败场景补充澄清问题。
- 对越界风险场景补充拒答和改写建议。
- 对高频模板进行版本迭代。
- 对低价值场景降级或隐藏。

## 8. 7d 观察协议

观察目标：确认 P0 是否具备扩大灰度价值，并筛出 P1 候选。

必看指标：
- 7 日留存。
- 周内重复使用用户占比。
- 用户最常用 5 类场景。
- 用户要求接入真实数据的频次。
- 用户要求自动执行的频次。
- 客服/运营收到的合规风险反馈。
- 模型成本与用量趋势。

决策输出：
- 继续灰度。
- 限制灰度并修复。
- 暂停灰度。
- 启动 P1 调研，但不混入 P0 完成口径。

每次验收、灰度观察或异常处理需记录：

| 字段 | 说明 |
|---|---|
| log_id | 唯一日志 ID |
| date | 日期 |
| reporter | 记录人 |
| role | 产品 / 技术 / 合规 / UI / 运营 |
| phase | P0-A 到 P0-H 或 24h / 72h / 7d |
| environment | dev / staging / review / production |
| build_id | App 构建号或版本 |
| model_policy_id | 模型白名单策略版本 |
| prompt_version | 提示词版本 |
| knowledge_pack_version | 知识包版本 |
| scenario | 场景类型 |
| input_type | text / speech_transcript |
| expected_result | 预期结果 |
| actual_result | 实际结果 |
| severity | P0 / P1 / P2 / P3 |
| boundary_type | product / technical / compliance / UI / model / data |
| evidence_link | 截图、录屏、日志、工单链接 |
| decision | pass / fail / watch / defer |
| owner | 责任人 |
| due_date | 处理期限 |
| follow_up | 后续动作 |
