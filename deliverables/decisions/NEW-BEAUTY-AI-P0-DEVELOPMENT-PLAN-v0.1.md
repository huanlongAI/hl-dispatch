# 新美业 AI 老板总助 App P0 开发计划

**文档编号**：NEW-BEAUTY-AI-P0-DEVELOPMENT-PLAN
**版本**：v0.1
**状态**：DRAFT v0.1
**日期**：2026-04-28
**市场**：中国大陆
**产品**：新美业 AI 老板总助 App
**Mode**：Mode B / Hardware-as-Substrate
**承载仓**：`hl-scene-app`（后续工程实现，不在本任务写入）
**设计系统**：`hl-scene-design-system` Flutter adapter
**首版 ICP**：1-3 家门店的单店/小连锁美业老板

**上游依据**：
- `tzhOS` R-0099 / AER-MODEL Mode B
- `hl-contracts` R-FE-CLIENT-001
- `hl-dispatch` DD-FE-CLIENT-v1
- `hl-contracts` R-036 amend-002（DS token-core + Flutter adapter + UniApp adapter）

---

## 1. 执行边界

本计划只定义 P0-A 到 P0-H 的阶段目标、交付物、验收门和依赖，不进入代码实现细节。

后续工程执行必须另行确认：
- 目标仓库。
- 分支策略。
- 写入权限。
- 契约依赖。
- CI 门禁。
- 发布通道。

本轮不写 `hl-scene-app`、`hl-scene-design-system`、`hl-contracts`、`hl-platform`。

## 2. 阶段计划

### P0-A：产品边界冻结

**目标**：
- 冻结 P0 用户、场景、输入、数据、硬件、执行和医疗边界。
- 明确“不接真实门店经营数据、不接硬件、不自主执行”的验收口径。

**交付物**：
- P0 产品规格。
- P0 非目标清单。
- 中国区模型策略。
- 风险与未决项清单。

**验收门**：
- Founder 或授权产品负责人确认 P0/P1 边界。
- 所有对外文案不得暗示真实系统接入、硬件接入或自动执行。

**依赖**：
- `tzhOS` R-0099 / AER-MODEL Mode B。
- `hl-dispatch` DD-FE-CLIENT-v1。

### P0-B：信息架构与 UI 规格

**目标**：
- 明确 App 页面、核心交互、状态、设计系统映射。
- 为设计师和 Flutter 工程提供统一 UI 输入。

**交付物**：
- P0 UI 设计规格。
- 页面流。
- 状态清单。
- 组件映射表。

**验收门**：
- 页面清单覆盖首页、对话、模板、历史、用量、设置、合规提示。
- 设计稿使用 `hl-scene-design-system` Flutter adapter。
- 语音转写、生成中、失败、拒答、用量不足等状态可验收。

**依赖**：
- `hl-scene-design-system` token-core + Flutter adapter。
- 后续 `hl-scene-app` 工程启动确认。

### P0-C：模型与提示词策略

**目标**：
- 建立中国区生产模型白名单与内部 benchmark 分离策略。
- 定义提示词模板、知识包、风险标签和回归用例。

**交付物**：
- 模型白名单草案。
- 提示词模板目录。
- 场景测试集。
- 合规拒答策略。

**验收门**：
- 生产依赖不包含 OpenAI / Claude / Gemini。
- 审核态与生产态模型路径一致。
- 轻医美相关输出只限经营、营销、客户沟通、门店管理和服务流程表达。

**依赖**：
- 中国区模型供应商合规确认。
- 法务/合规审核口径。

### P0-D：系统架构与数据边界

**目标**：
- 固化 App、API Gateway、模型路由、知识模板、审计、用量边界。
- 明确 P0 与 P1 数据分界。

**交付物**：
- P0 系统架构。
- 数据流说明。
- 审计字段建议。
- 用量/积分/订阅边界。

**验收门**：
- 架构图不包含真实预约、客户、排班、库存、营销系统接入。
- 不包含智能眼镜、录音卡、持续录音链路。
- 任一生产回答可追溯模型版本、提示词版本和知识包版本。

**依赖**：
- 账号与用量服务边界确认。
- 审计日志规范确认。

### P0-E：内容与模板准备

**目标**：
- 准备新美业老板高频经营场景模板。
- 将模板做成可评测、可复核、可版本化资产。

**交付物**：
- 活动方案模板。
- 私域文案模板。
- 客诉回复模板。
- 员工培训 SOP 草稿模板。
- 复盘提纲模板。
- 风险表达禁区清单。

**验收门**：
- 模板可在无真实门店数据条件下运行。
- 模板不得包含医疗诊断、治疗建议、处方、疗效承诺。
- 每个模板至少有 3 条回归用例。

**依赖**：
- 产品场景优先级。
- 合规审核口径。

### P0-F：工程实施准备

**目标**：
- 为后续 `hl-scene-app` 工程执行准备分支、任务、接口、验收口径。
- 不在本阶段直接进入实现，除非另行确认目标仓与权限。

**交付物**：
- 工程任务拆分。
- 依赖清单。
- 目标仓写入授权确认。
- CI 验收建议。
- 发布前检查表。

**验收门**：
- 明确不写非授权仓库。
- 目标仓、分支、负责人、验收命令确认。
- 与设计系统、后端服务、模型服务、合规审查的依赖明确。

**依赖**：
- Founder 或授权 owner 确认工程启动。
- `hl-scene-app` 当前分支状态 clean/synced。

### P0-G：灰度验收准备

**目标**：
- 设计产品、技术、合规、UI 验收方法。
- 建立 24h/72h/7d 观察协议。

**交付物**：
- P0 验收观察文档。
- 灰度问题分级。
- Air Dispatch Log 模板。
- 回滚和降级口径。

**验收门**：
- 验收项覆盖产品、技术、合规、UI。
- 观察指标可记录、可复盘。
- 不把产品上线等同于产品实现已完成。

**依赖**：
- P0-A 到 P0-F 的输入。
- 运营、客服、合规参与。

### P0-H：发布决策与复盘

**目标**：
- 汇总验收与观察结果，形成是否进入更大范围灰度的决策输入。
- 收敛 P1 需求，不让 P1 扩展污染 P0 验收。

**交付物**：
- P0 验收报告。
- 观察期复盘。
- P1 候选需求清单。
- 风险关闭或延期清单。

**验收门**：
- 所有 P0 阻断项关闭或有明确延期裁决。
- 合规、模型、UI、技术验收有签字或等价记录。
- P1 需求全部标为候选，不混入 P0 完成口径。

**依赖**：
- 灰度数据。
- 用户反馈。
- 审计与用量日志。

## 3. 总体验收依赖

- 产品边界冻结后才能进入工程实施准备。
- UI 规格冻结后才能进入 Flutter 页面实现。
- 模型白名单确认后才能进入中国区生产调用链。
- 合规拒答策略确认后才能进入应用市场审核材料准备。
- 目标仓与权限确认后才能写入 `hl-scene-app`。
