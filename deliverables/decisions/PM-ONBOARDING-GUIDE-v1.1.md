# PM 接入指南

> 版本：v1.1 | 日期：2026-04-02
> 适用对象：邹骢（商品与销售线）、朱阳（客户与支付线）

---

## 写在前面

你已经在 Cowork 中推进了一些工作，这不是从零开始。这份指南的目的是把你已有的进展正式接入创始人的仓库和治理流程，让你的工作成果进入正式交付管线。

接入之后，你的日常工作在 Cowork 中完成，AI 全程参与你的思考、写作和编码。你是产线责任人——从规格定义到 AI 编码到测试验收，整条线由你驱动。


---

## 一、你的角色

你是所负责能力包的「产线责任人」。

**你负责的事：**
- 接入创始人已定义的契约体系和治理规则（上游）
- 编写能力规格书（Cap-Spec）：定义这个能力包做什么、怎么判定做对了
- 用 AI 完成代码实现（工具以 Cowork 为主，可辅以 CLI）
- 编写测试用例，驱动 AI 生成测试代码
- 基于验收场景逐条验证
- 交付给技术验收官（李旭阳）审计

**不是你的事：**
- 修改契约体系的根定义（那是创始人的事，你可以提 Issue 建议）
- 在自己能力包范围之外做决策
- 发明不在契约中的治理规则
- 跳过技术验收直接上线

## 三、接入步骤

你已经有一些基础，所以时间估算按实际情况压缩。没有接触过的部分认真做，已经熟悉的部分快速过一遍确认即可。

### Phase 1：知识体系对齐（Day 1-2）

目标不是从头学习，而是确认你已有的理解和创始人的治理体系一致。在 Cowork 中打开对应文件，让 AI 帮你解读和验证理解。

| 步骤 | 文档（hl-contracts 仓库内）| 你需要确认什么 | 预计时间 |
|------|--------------------------|---------------|----------|
| 1 | `CONTEXT.md` | 系统世界观、5 条冻结不变量 | 30min |
| 2 | `governance/SAAC-HL-001-v1.1.md` | 架构宪法：3 域 × 3 态、7 常量层 | 30min |
| 3 | `prd/README.md` | PRD 分区规则：core 禁 UI，console 禁自创规则 | 15min |
| 4 | `rules/GOVERNANCE-PATTERN.v1.0.md` | Can → Action → Audit 三步闭环 | 15min |
| 5 | `rules/biz-capabilities-blueprint.yaml` | 你负责的能力包 key_action，标记你认为需要调整的点 | 15min |
| 6 | 选一个已有 core PRD 阅读 | Cap-Spec 的章节结构和引用方式 | 30min |
| 7 | `reasoncodes/reasoncodes.csv` 前 50 行 | reason_code 命名规则 | 10min |
| 8 | `INDEX.md` | 能快速定位仓库中任意文件 | 10min |

**完成标志：** 在飞书 #PM工作台 群发一条消息，用自己的话概括：
- 你负责的能力包有哪些 key_action，哪些你认为需要调整
- Can → Action → Audit 闭环是什么意思
- core PRD 和 console PRD 的铁律分别是什么

创始人会查看这条消息作为 Phase 1 的检查点。

### Phase 2：环境接入（Day 1-2，15 分钟）

可以和 Phase 1 并行完成。

| 步骤 | 做什么 | 怎么验证 |
|------|--------|---------|
| 1 | 把 hl-contracts 仓库 clone 到本地（如已有则 pull 最新）| `git pull` 成功 |
| 2 | 创建一个测试分支并 push | push 成功后删除测试分支 |
| 3 | 打开 Cowork，确认 `.claude/CLAUDE.md` 已配置 | AI 能回答「我负责哪些能力包」|
| 4 | 确认飞书通知畅通 | 在 #工程通知 看到自己的 test push 通知 |

**完成标志：** 截图发给创始人：Cowork 中 AI 正确回答了你的角色和职责。

---

### Phase 3：首个 Cap-Spec 编写（Day 2-5）

以你的首个能力包为对象，整合你已有的业务理解，按正式模板写出 Cap-Spec v1.0。

**在 Cowork 中工作。** 让 AI 帮你参照模板起草、检查引用、预跑校验。

| 步骤 | 做什么 | 预计时间 |
|------|----------|----------|
| 1 | 读上游契约：找到与你能力包相关的 decisions/ + rules/ + facts/ | 1h |
| 2 | 在 Cowork 中编写 Cap-Spec 草稿（整合你已有的业务理解）| 1.5h |
| 3 | 本地跑 prd-gate 检查（AI 可协助）| 15min |
| 4 | 提交 PR → 自动触发 claude-gate + prd-gate CI | 15min |
| 5 | 根据 CI 反馈修改，请求创始人 Review | 1h |

Cap-Spec 文件命名：`Cap-Spec-Biz.{Module}.v1.0.md`
存放位置：`hl-contracts/prd/core/`

**完成标志：** PR 提交成功，claude-gate + prd-gate CI 通过，创始人收到 Review 请求。

---

### Phase 4：契约补齐（Day 5-10）

Cap-Spec PR 合并后，按以下顺序补齐契约文件。顺序不可调换，后续文件依赖前序。

| 顺序 | 工件 | 存放位置 | 预计时间 |
|------|------|---------|---------|
| 1 | Facts 定义 | `facts/facts-catalog.md` 新增你的能力包分区 | 2h |
| 2 | Decisions 合同 | `decisions/biz.{module}.*.can.v1.md` | 3h |
| 3 | ReasonCodes | `reasoncodes/reasoncodes.csv` 新增行 | 1h |
| 4 | API 契约 | `apis/biz.{module}.internal.openapi.v1.yaml` | 2h |

每个文件单独提 PR，不要混在一起。

**完成标志：** 4 个 PR 全部合并，能力包在 hl-contracts 中有完整的契约定义。

---

### Phase 5：AI 编码 → 测试 → 验收（Day 10+）

进入持续交付节奏。

| 环节 | 你做什么 | AI 做什么 |
|------|---------|----------|
| 编码 | 描述需求、审查生成结果 | 生成代码实现 |
| 测试 | 编写测试场景、验收标准 | 生成测试代码 |
| 验收 | 逐条验证验收场景 | 协助运行和分析结果 |
| 提交 | PR + 请求验收官审计 | 协助 CI 反馈修复 |

Cowork 为主要工作环境，需要跑代码时可以用 CLI。

**完成标志：** 技术验收官（李旭阳）审计通过，创始人里程碑签收。

## 四、能力包审查（接入期间同步推进）

当前 10 个能力包定义（biz-capabilities-blueprint.yaml）源自早期体系推导。在你接入的过程中，请同步完成以下审查：

1. **边界合理性**：你负责的每个能力包，边界是否清晰？是否有重叠或遗漏？
2. **key_action 完整性**：列出的 key_action 是否覆盖了真实业务场景？哪些需要增减？
3. **能力包拆分/合并**：是否有能力包应该拆分或合并？

审查结果通过 Issue 提交（标记 `decision-request`），创始人裁决后更新蓝图。

## 五、日常工作节奏

```
上午                         下午                      收尾
────────────────────────────────────────────────────────────
飞书通知 + Review             Cowork AI 编码/测试        落地 + 提交
├ #PM工作台 看 review 通知     ├ 基于 Cap-Spec 驱动 AI    ├ 代码 PR / Cap-Spec PR
├ #工程通知 看 CI 状态          ├ 修复 CI 失败            ├ 更新 Issue 状态
├ GitHub PR review 评论        └ 补充验收场景             └ 飞书讨论结论写入 GitHub
└ 创始人裁决回复

周一：#任务协同 对齐本周目标
周五：更新 Bitable 看板 + 关闭已完成 Issue
```

**核心原则：PM 不等人。** 创始人给定上游后，你自主驱动。遇到上游问题提 Issue 推动，不原地等待。

## 六、铁律（6 条）

1. **上游不可改** — 契约体系的根定义你不能直接改，需走 PR + 创始人审批
2. **PRD 分区铁律** — core PRD 禁止写 UI，console PRD 禁止自创业务规则
3. **治理规则不可自创** — 不得在能力包内发明不在契约中的规则
4. **技术验收不可跳过** — 交付物必须经技术验收官审计
5. **产线不可越界** — 只在自己负责的能力包范围内行使驱动权
6. **讨论 24h 落地** — 飞书中的讨论结论须在 24h 内写入 GitHub Issue/PR

## 七、AI 协作的关键要求

接入即启动 AI 驱动的全流程工作方式。不是“用 AI 帮忙”，而是“AI 全程参与”。

1. **所有规格编写在 Cowork 中完成** — AI 能看到你写的每一行，能实时检查引用和规范
2. **飞书讨论结论必须落地到 GitHub** — 只有写进 Issue/PR 的内容，AI 在下次会话中才能读到。不落地的讨论等于没发生
3. **遇到问题先问 AI** — AI 知道契约体系的全貌，能告诉你该引用哪个文件、该走什么流程
4. **每次 Cowork 会话结束前** — 让 AI 帮你总结本次进展，列出下次要做的事
5. **任务和决策必须 AI 可追溯** — 不要在微信/电话中做决策然后不记录。所有沟通结论要么在 Cowork 中产生（AI 自然知情），要么写入 GitHub（AI 可检索）

AI 不是打字机，是你的搭档。你描述意图，AI 帮你执行和检查。

## 八、遇到问题怎么办

| 情况 | 做什么 |
|------|--------|
| 不理解某个契约文件 | 在 Cowork 中让 AI 解读 |
| 上游契约缺失或矛盾 | 在 hl-dispatch 提 Issue，标记 `decision-request` |
| 能力包定义需要调整 | 提 Issue 说明调整理由，等创始人裁决 |
| CI 不通过 | 让 AI 分析 CI 日志并协助修复 |
| 架构问题（跨能力包） | 在 #PM工作台 讨论，24h 内落地到 Issue |
| 被阻塞超过 1 天 | 在 #任务协同 提出，不要沉默等待 |
