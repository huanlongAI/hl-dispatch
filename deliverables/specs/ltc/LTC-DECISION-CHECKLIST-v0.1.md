# LTC 待裁决清单 v0.1

**状态**：已回写创始人裁决；待进入 R1 子规格展开
**日期**：2026-05-06
**负责人**：NODE-A
**范围**：LTC-Endpoint R0/R1 进入实现前的待裁决问题

## 1. 目的

本文把 LTC R0/R1 规格中的模糊点压缩为可直接裁决的问题清单，并记录创始人已确认口径。

裁决完成前，LTC 不进入 runtime、真实员工数据、飞书台账、飞书公告或绩效集成实现。

## 3. 逐项裁决

### C1 强制部署设备范围

创始人裁决：

- LTC 强制部署范围仅限公司固定资产电脑。
- 平台优先级为 Windows 优先，再 Mac。
- LTC 仅支持 Windows / macOS 双平台，不支持 Linux、移动端或其他平台。

不建议：

- 直接覆盖所有个人设备。
- 让员工本地自行决定关闭或卸载。

影响：

- 影响 R3 Endpoint Sentinel Alpha 的设备池。
- 影响用工治理、设备管理和员工确认文本。

### C2 BYOD / 个人设备

创始人裁决：

- 员工个人电脑不纳入强制范围。
- 员工自愿选择可安装，不作制度规定。
- 自愿安装仍必须遵守 LTC 的 banned 字段、安全和下游边界。

影响：

- 避免把企业基础设施边界扩张到未裁决的个人空间。

### C3 员工状态路径

创始人裁决：

- 安装前置工作由行政部门处理完毕。
- LTC 接入视为前提合规流程已完毕。
- 未处理完成的员工或设备不会进入安装环节。
- LTC runtime 不处理拒签、未确认、离职、外包、试用期等用工前置流程。

影响：

- 影响 SystemAcknowledgementRecord、RetentionPolicy 和管理员操作审计。

### C4 首个平台

创始人裁决：

- 见 C1。
- R2 不绑定平台。
- R3 runtime 优先 Windows 公司固定资产电脑。
- Mac 作为第二平台进入。
- 只支持 Windows / macOS 双平台。

影响：

- Windows 需要优先明确常驻服务、签名、开机自启、健康检查和管理员卸载流程。
- Mac 作为第二平台，需要明确 launch daemon、签名、开机自启、健康检查和管理员卸载流程。

### C5 MDM / 受管安装

创始人裁决：

- 暂时不接入 MDM 或类似平台。
- 公司资产由公司分配给员工使用。
- 设备由人工指定和登记。
- LTC 只采集当前安装电脑的硬件配置信息。
- LTC 不参与资产管理。

影响：

- 影响 R3/R4 的安装台账、设备绑定和人工登记流程。

### C6 字段范围先行

创始人裁决：

- 字段可以先行。
- 在采集范围内尽量齐全。
- 字段齐全不改变 banned 字段永久禁止边界。
- 所有字段必须写入字段范围版本和员工可见说明。

采集前必需材料：

- 字段目的。
- 最小化证明。
- banned 泄漏测试。
- 制度版本升级。
- 员工重新确认。
- 回滚方案。

### C7 raw observation 生命周期

创始人裁决：

- R2 只使用 synthetic fixture。
- R3 后 raw observation 只允许作为内存候选输入。
- raw observation 生命周期限制在单次清洗流程内。
- 进程崩溃、debug、metrics、日志、快照都不得包含 raw observation。

影响：

- 影响技术设计、测试矩阵和日志规范。

### C8 Tool Source Registry 首个工具

创始人裁决：

- R2 首批工具来源为 Codex 订阅和 `heiyucode.com` API 密钥使用。
- 工具来源必须登记工具名、版本、采集方式、字段映射、泄漏测试和审批记录。

待补问题：

- Codex 订阅使用能取得哪些结构化元数据。
- `heiyucode.com` API 密钥使用能取得哪些结构化元数据。
- 若工具本身不提供可靠 token / request 元数据，R2 不做推断。

### C9 evidence feed 周期与粒度

创始人裁决：

- R5 前定义 evidence feed 批次周期，默认按自然日或绩效周期切片。
- 每个批次必须包含 `batchId`、期间、制度版本、字段范围版本、确认版本、签名和 schema 版本。
- 重试必须幂等，不能重复记账。
- evidence reference 只引用清洗证据，不暴露 raw observation。

影响：

- 影响大辉子消费、飞书台账引用和审计追踪。

### C10 留存与访问

创始人裁决：

- R3 前定义留存期限。
- 创始人、大辉子、管理员、审计者、员工 owner view 分别定义可见粒度。
- 导出默认禁止，除非审计或创始人批准。
- 申诉或审计期间进入冻结，不执行删除。

待补问题：

- 具体留存期限。
- 申诉冻结最长时间。
- 审计冻结解除条件。

### C11 Owner View 与申诉 SLA

创始人裁决：

- 员工可看字段类别、制度版本、确认版本、端点状态、断链记录、说明状态和申诉入口。
- 员工说明以 amendment / explanation 关联 evidence，不改写已签名 evidence。
- 申诉 SLA 需要单独定义，例如响应时限、处理人、升级路径。

影响：

- 影响员工信任、合规解释和飞书公告前置条件。

### C12 飞书台账与公告

创始人裁决：

- 飞书台账和公告只由大辉子执行。
- LTC 无飞书写权限。
- 公告前必须校验证据引用、制度版本、员工说明状态、可见范围、敏感信息排除和必要审批。
- R4/R5 dry run 不进入正式台账和公告。

待补问题：

- 公告范围是全员、管理层、当事人，还是按制度类别分级。
- 哪些场景需要创始人审批。

### C13 break-chain 绩效使用

创始人裁决：

- break-chain 只作为证据完整性、端点健康和合规状态输入。
- 大辉子可在制度范围内分析 break-chain，但不能把单次断链自动解释为绩效负面。
- 员工说明必须可关联到 break-chain。

影响：

- 防止 LTC 被误解为间接纪律或绩效处罚器。

### C14 旧 P0 文档废止方式

创始人裁决：

- 如能安全删除旧 P0 文档，则优先删除。
- 如不能安全删除，则建立 LTC active baseline 索引。
- 不能删除的旧 P0 文档首页逐个标注 Historical Only / Superseded。
- 旧文档中永久保留 banned 字段边界和 LTC 不直接绩效裁决边界。
- 废止旧“不用于 HR / 绩效”“员工可以停止 endpoint agent”“不安装 runtime”等口径。

影响：

- 防止后续执行继续引用旧 P0 造成漂移。

## 4. 建议裁决顺序

1. C1-C5：已锁设备、平台、安装和资产边界。
2. C6-C8：已锁字段先行、raw observation 和首批工具来源。
3. C9-C13：已锁 evidence feed、留存访问、owner view、飞书门禁和 break-chain 解释。
4. C14：待执行旧 P0 文档安全删除评估。
