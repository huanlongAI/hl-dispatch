01ORG · POWER

组织结构与四权分离23 人 · 6 角色 · 1 创始

唤龙采用「四权分离 (Four-Power Separation)」治理模型：将语义主权、技术实现权、独立质量权、组织治理权分别授予不同角色，任意单一角色均无法独立完成「需求 → 实现 → 验收」闭环。Founder（@tongzhenghui）保留组织治理权与最终合并权，但不直接参与功能编码。

04GATE · L1—L4

05FLOW · 12 STATES

五步交付流程 + 12 状态机单向状态推进 · 失败回环

每个功能从「Proposed 提案」流到「Released 发布」需穿过 12 个治理状态、由 6 个角色协同推进。任意门禁失败均回环至最近一个可修复的角色泳道，而不是回到起点。

STEP 01
Cap-Spec 冻结
Owner: Founder + PM
在 hl-contracts 写入能力规约（行为 / 不变量 / 边界），由 Founder 二人评审签发 contract-locked tag。
STEP 02
Design + HPRD
Owner: PM + Designer
PM 撰写 HPRD（高保真 PRD），Designer 产出 DTCG token 与 Widgetbook 组件，pm-hprd-pass 通过后冻结。
STEP 03
AI-First 实现
Owner: Engineer
Engineer 以 AI 协助方式编码，强制 TDD（先写 L2 测试 → 再实现），Draft PR 持续推送，触发 L1 + L2 反馈循环。
STEP 04
门禁 + QA
Owner: CI + QA
L1 → L2 → L3 顺序执行；QA 设计的 DD-TEST 行为场景跑通 → 签发 qa-verdict。
STEP 05
PM 验收 + 合并
Owner: PM + Founder
PM 执行端到端 HPRD 验收 → pm-acceptance 通过 → Founder squash & merge → release-train 自动产出版本。

① 外部消息队列
✕
禁用 Kafka / RabbitMQ / RocketMQ / Pulsar — 由 Spring Modulith Event Publication Log + Outbox 替代，零外部依赖。
解锁条件：跨 service 异步 + 持久化吞吐 ≥ 5k tps
② Redis / Cache 层
✕
禁用 Redis / Caffeine / EhCache — Phase 0 数据规模 PostgreSQL 18 + index 完全覆盖；缓存引入需 SLO 证明。
解锁条件：p95 query > 200ms 持续 14 天
③ Keycloak / 外部 IdP
✕
禁用 Keycloak / Auth0 / Casbin — 使用 Spring Security 内置 JWT + 应用级 RBAC，避免运维负担与外部锁定。
解锁条件：需要联合身份 / SAML / 第三方登录
④ Nacos / Apollo 配置中心
✕
禁用任何外部配置中心 — application.yml + Spring Profiles + GitOps 静态发布；动态配置在 Phase 0 是过度工程。
解锁条件：多区域 + 实时灰度需求
⑤ gRPC / Thrift
✕
禁用 gRPC / Thrift / Protobuf RPC — 仅 OpenAPI 3 + REST/JSON；服务边界由 Spring Modulith 模块化解决，不依赖跨进程协议。
解锁条件：跨语言服务 + 流式 RPC 必要
⑥ JOOQ / MyBatis
✕
禁用所有非 JPA 数据访问层（H-002 BLOCKING）— 维持单一 ORM 心智模型，避免 schema 漂移与 N+1 调试矩阵爆炸。
解锁条件：无（H-002 永久 BLOCKING）
⚠ Kubernetes
⏳
DEFERRED — Phase 0 用 systemd + docker compose 即可承载流量；K8s 引入需 CapEx + 运维 headcount 评估。
解锁条件：≥ 3 节点常驻 + 自动扩缩需求
⚠ Terraform / IaC
⏳
PENDING（J-01）— 当前云资源由脚本声明；待 J-01 裁决签发后切换至 Terraform 单源管理。
解锁条件：RULING J-01 签发
