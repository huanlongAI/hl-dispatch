二、P0 产品边界

这次明确支持

  • 中国大陆市场的新美业老板，首版 ICP 是 1-3 家门店的单店/小连锁经营者。
  • 文字输入 + App 内主动语音转写。
  • 活动方案、私域文案、客诉回复、员工培训 SOP、复盘提纲等草稿型输出。
  • 收藏、历史、复制、分享、用量/订阅入口等 App 壳能力。

这次明确不做

  • 不接真实预约、客户、排班、库存、营销、财务系统。
  • 不接智能眼镜、录音卡、持续录音、后台录音。
  • 不自动群发、不自动改价、不自动改库存、不自主执行。
  • 不提供医疗诊断、治疗建议、处方、疗效承诺。
P0 / P1 隔离是这批文档的核心
P1 可评估多角色协作、真实门店数据接入、硬件输入、自动化执行，但这些都不能反向污染 P0 验收口径。站点在描述时必须继续保留这种“能力可想象，当前不承诺”的边界。

三、UI 与交互要点

维度P0 要点
承载栈P0 原生 App 承载在 hl-scene-app，只消费 hl-scene-design-system Flutter adapter；若后续切小程序入口，则走 UniApp adapter。两条 runtime 都不得复制或绕开 token-core。
核心页面启动/加载、登录/授权、首次引导、首页、AI 对话页、模板中心、草稿详情、收藏/历史、用量/订阅、设置、合规提示页。
关键状态录音中、转写中、生成中、生成失败、用量不足、网络异常、合规拒答、未登录、账号异常。
交互重点语音只在用户主动触发时录入；转写文本必须可编辑后再发送；输出强调“人工复核”，不暗示系统已执行动作。
视觉边界专业、清晰、轻运营工具感；遵循 Design System token，不新增未登记品牌色，不制造“已接入真实系统/硬件”的视觉暗示。

如果你只看一条：No manual magic。这版 UI 不是“更炫的聊天页”，而是“低输入门槛 + 可复制草稿 + 明确边界提示”的经营助理工作台。

四、系统与模型策略

1

App Shell

Flutter App 承载登录、对话、模板、草稿与本地缓存，不直接绑定门店经营系统。

2

Input Layer

文字输入与主动语音转写进入同一请求流；原始音频默认不作为长期业务数据。

3

Policy Layer

Prompt Registry、知识包、合规边界、澄清问题、安全改写都在服务端策略层统一编排。

4

Model Router

生产只走中国区合规白名单；OpenAI / Claude / Gemini 仅用于内部 benchmark，不进公开生产链路。

Prompt Registry 最少需要的层

  • global_guardrail
  • role_prompt/<entry> 四类入口
  • safety_rewrite
  • clarifying_question
  • output_formatter
  • route_classifier

审计必须保留的字段

  • trace_id、场景类型、模型版本、提示词版本、知识包版本
  • 风险标签、token 用量、模型供应商、创建时间
  • 审核态与生产态需走同一路径，便于抽样核验
  • 禁止长期明文保留用户粘贴的敏感个人信息

验收最重要的四条硬边界：

  • 不依赖真实门店经营系统数据。
  • 不依赖智能眼镜、录音卡、持续录音或后台录音。
  • 不出现自主执行能力。
  • 中国区生产态与审核态必须使用同一白名单模型路径。

六、这对站点意味着什么

  • 产品团队页 现在可以用一个具体样板解释 Cap-Spec、边界冻结和 pm-acceptance，而不只停留在抽象流程。
