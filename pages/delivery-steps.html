第一步 — Cap-Spec 冻结

第一步：Cap-Spec 冻结（语义冻结）

产出物
  • Cap-Spec 文档
  • 功能定义清单
  • 创始人裁决签字
参与角色
  • PM 编写 Spec
  • 创始人审核
  • 工程师提供技术可行性反馈
通过条件
  • Spec 定义清晰完整
  • 创始人确认同意
  • 工程师确认技术可行
下一步门禁
  • 等待 HPRD 准备
  • 设计评审启动
失败处理
  • 返回 PM 修改
  • 重新提交创始人审核
状态
  • Proposed
  • Cap-Spec Frozen

第二步：设计与理解确认（Design + HPRD）

产出物
  • 设计稿 / Design Packet
  • 工程师 HPRD 文档
  • QA 验收测试清单
参与角色
  • 设计师出设计
  • 工程师编写 HPRD
  • QA 编写验收测试
  • PM 确认理解
通过条件
  • HPRD 与 Cap-Spec 一致
  • PM 确认理解无误
  • QA 验收测试已编写
  • 技术方案评审通过
下一步门禁
  • 开始实现阶段
  • 分配工程师任务
失败处理
  • 返回工程师修改 HPRD
  • 设计重审
  • 若理解有偏差，需要回到 Cap-Spec
状态转移
  • HPRD In Review
  • HPRD Approved
  • Design Frozen

第三步：实现（Implementation）

产出物
  • PR 提交
  • 代码实现
  • 提交信息与文档
参与角色
  • 工程师 AI-first 编码
  • 工程师审查 AI 输出
  • 架构评审（可选）
通过条件
  • 实现基于 HPRD
  • 代码质量达标
  • AI 输出经过人工审查
  • PR 链接到需求
下一步门禁
  • 提交到远端 PR
  • 触发 CI/CD 流程
失败处理
  • 本地修复，继续迭代
  • 重新提交 PR
  • 严重偏离 HPRD 则返工
状态转移
  • In Development
  • PR Submitted

第四步：门禁与验收（Gates + Acceptance）

产出物
  • CI/CD 门禁通过报告
  • QA qa-verdict
  • 缺陷列表（若有）
参与角色
  • CI/CD 系统自动门禁 L1-L4
  • QA 手动测试与出 Verdict
  • 工程师修复缺陷
通过条件
  • 单元测试通过
  • 集成测试通过
  • 代码风格检查通过
  • 安全扫描通过
  • QA 手动测试通过
  • QA Verdict = Approved
下一步门禁
  • 进入 PM 验收阶段
  • 准备合并到主分支
失败处理
  • 门禁失败 → 工程师修复
  • QA Verdict 拒绝 → 返回工程师
  • 反复失败 → 考虑返回设计阶段
状态转移
  • In Review
  • QA Verdict
  • 门禁失败 → 返回修复

第五步：产品验收与合并（PM Acceptance + Merge）

产出物
  • PM pm-acceptance 签字
  • Merge 到 main 分支
  • 版本发布
参与角色
  • PM 最终验收
  • 工程师完成 merge
  • DevOps 负责发布
通过条件
  • 所有前置门禁通过
  • 功能符合 Cap-Spec
  • PM 确认验收
  • 所有缺陷已解决或记录
下一步
  • Merge to main
  • 自动或手动发布到生产
  • 标记版本 v.x.x
失败处理
  • PM 拒绝 → 返回工程师修复
  • 严重偏离需求 → 回到设计评审
最终状态
  • PM Acceptance
  • Merged
  • Released

平行开发轨道

后端（hl-platform）和前端（hl-app）可以在共享 Cap-Spec 的基础上并行开发。

后端轨道（hl-platform）

  1. Cap-Spec Frozen — 共享定义
  2. HPRD for Backend — 后端工程师编写 API 设计、数据模型、业务逻辑说明
  3. Backend Implementation — 后端编码、API 测试
  4. Integration Testing — 与前端集成测试
  5. PR → QA Verdict — 后端 PR 通过 QA 测试

前端轨道（hl-app）

  1. Cap-Spec Frozen — 共享定义
  2. Design Packet + HPRD for Frontend — 前端设计稿、交互规范、API 调用说明
  3. Frontend Implementation — 前端编码、单元测试
  4. Integration Testing — 与后端集成测试
  5. PR → QA Verdict — 前端 PR 通过 QA 测试

同步点与合并

  • 同步点 1：Cap-Spec — 两条轨道从同一份产品定义出发
  • 同步点 2：集成测试 — 后端 API Ready 后，前端可以集成测试
  • 同步点 3：pm-acceptance — PM 对两条轨道的共同功能进行验收
  • 同步点 4：Release — 后端和前端同时发布到生产环境（或按需序列化发布）

如果后端或前端轨道卡住，需要启动 escalation 流程，由创始人决定是否延迟发布、分阶段发布或调整范围。

五步交付流的核心原则

  • 串行控制： 每一步都有明确的入口和出口条件，前一步未完成不能进入下一步。
  • 质量门禁： CI/CD 和 QA 测试是自动化的质量关卡，不受人为压力影响。
  • 角色分工： PM、工程师、QA、创始人各有各的验收职责，互相制约。
  • 并行优化： 后端和前端可以基于共同的 Cap-Spec 并行开发，提高交付效率。
  • 可观测性： 每个状态转移都有记录，便于追踪进度和问题分析。
  • 降级策略： 如果后续步骤发现问题，有明确的降级和修复路径。
