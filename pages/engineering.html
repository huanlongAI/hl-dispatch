💻 工程师团队指南

唤龙平台 AI 驱动协作模型
后端 8 人 + 前端 5 人

适用角色：后端工程师（8 人，含 2 名 PHP）+ 前端工程师（5 人） 关联文档：全局工作流 · 前端与设计指南 · 测试团队指南 治理来源：TEAM-COLLABORATION-SPEC v2.1 §2.3 / §5.2 / §7 · TOOLCHAIN-GUIDE v1 · UI-DELIVERY-SPEC v1.0

一、你在新模型中的角色定位

工程师 — 方案 Owner（Solution Owner）

你拥有技术方案设计权和实现权。在 AI-first 模式下，AI 生成代码为主，你负责：

  • 意图定义：定义类型、接口、组件 API、状态结构
  • 审查裁决：Review AI 产出，判断方案是否正确
  • 必要补充：在 AI 覆盖不到的地方手动编码
  • 统一过 gate：不论代码来源（AI 生成或手写），所有 diff 统一经过 CI 门禁

你拥有的：

  • 技术方案设计权（design.md）
  • AI-first 编码实现权
  • 单元测试、集成测试编写权
  • HPRD 编写权（Cap-Spec 的理解确认件）
  • 代码互审权（Code Review）

你不做的：

  • 不定义业务需求（Cap-Spec 是 PM 的 SSOT）
  • 不做最终质量放行（qa-verdict 是 QA 的独立权力）
  • 不修改治理规则（交创始人）

三、HPRD 怎么写

HPRD = Human-readable Product Design，是你对 PM 的 Cap-Spec 的理解确认件。它证明"你理解对了"。

最小模板

字段 必填 你写什么
Capability ID 是 对应哪个能力包
User-visible Changes 是 用户会看到什么变化（用非技术语言描述）
Unchanged Behavior 是 哪些行为必须保持不变
Out of Scope 是 这次明确不做什么
Acceptance Walkthrough 是 PM 将如何验收（对应 Cap-Spec-2 的 Case-ID）
Semantic Questions 否 你理解中仍未决的问题

注意事项

  • HPRD 不是上游规格——上游 SSOT 是 PM 的 Cap-Spec
  • 如果 HPRD 和 Cap-Spec 冲突，以 Cap-Spec 为准，你修正 HPRD
  • HPRD 不包含技术实现细节（数据库设计、API 内部结构、类层次等）——那是 design.md 的事
  • PM 审批 HPRD 只看一件事："业务语义理解对不对"
  • pm-hprd-pass（流程内检查）未通过前，你不得开始 Step 3 实现

四、能力包流中你的完整时间线

Step 1: PM freeze Cap-Spec → 你等待
    ↓
Step 2: 你开始
  ├── 阅读 Cap-Spec
  ├── 写 HPRD → 交 PM 审批
  ├── 写 design.md → 交工程师互审 + AI reviewer
  └── （QA 同步编写 acceptance-manifest）
    ↓
  ⛔ pm-hprd-pass 必须通过
  ⛔ design.md 互审必须通过
    ↓
Step 3: 你开始实现
  ├── AI-first 编码：功能代码
  ├── 单元测试（governedAction 五条必过 + domain）
  ├── 集成测试（Testcontainers）
  └── 可运行 demo
  → 提交 Draft PR
    ↓
Step 4: 门禁 + 验收（并行）
  ├── CI 门禁 L1-L3 自动执行（你需要确保全绿）
  ├── L4 AI 审计（Advisory，参考但不阻塞）
  ├── QA 独立验收 → qa-verdict
  └── 工程师互审 Code Review
  → Draft PR 转 Ready for Review
    ↓
Step 5: PM 产品验收 → pm-acceptance → Merge

五、PR 提交规范

每个 PR 必须在 body 中声明分类字段：

change_class: capability    # capability | bugfix | tech-improvement | governance
contract_touch: false       # true | false
founder_required: false     # true | false

自动升级规则：

  • contract_touch = true → founder_required 必须 true
  • change_class = governance → founder_required 必须 true
  • change_class = capability 且 contract_touch = false → CI 自动放行，不需要创始人审批

Contract Touch Checklist（当 contract_touch = true 时）：

contract_touch_detail:
  reason_code_change: false
  iron_rule_impact: false
  api_contract_change: false
  event_schema_change: false
  cross_tenant_impact: false

六、CI 门禁——你需要过哪些关

L1 格式性（BLOCKING）

编译通过、类型检查、代码格式（ktfmt/dart format）、License Header、YAML 语法

L2 确定性（BLOCKING）

契约一致性、模块边界、ReasonCode 对齐、循环依赖检测、API 路径规范、G-026 验收覆盖率（YAML 解析）、Spec 漂移检测

L3 行为性（BLOCKING）

单元测试全过、集成测试全过（Testcontainers）、G-023 验收场景回放、DoD 静态验证、回归集

L4 语义性审计（ADVISORY）

双模型交叉审计（GPT + Claude）、意图一致性、代码可读性、安全审计（P0 安全问题会升级为 BLOCK）

铁律：L1-L3 全绿是 merge 的硬前提。没有例外。

七、后端工程师：Java → Kotlin 适配

技术栈：Kotlin 2.1.10 / Spring Boot 3.5.11 / Spring Modulith 1.3.4

在 AI-first 模式下的适配路径：

  • AI 生成 Kotlin 代码，你需要读懂它（sealed class、value class、coroutines）
  • 你对 JVM 生态的理解（Spring Boot / JPA / Gradle）可以直接复用
  • 通过实际能力包开发在 AI 编码过程中自然习得 Kotlin 特性
  • 你的核心价值在于：定义正确的类型和接口 → 让 AI 在约束内编码 → Review AI 产出

八、前端工程师的额外说明

前端工程师除了遵循本指南外，还需要阅读 前端与设计团队指南，特别关注：

  • 双线交付模型（Foundation Lane + Capability UI Delivery Lane）
  • 硬规则：No UI Input Ready, No Frontend Coding——等 design-ready AND api-mock-ready 双就位
  • UI 风险分层（UI-R0 ~ R3）和对应门禁
  • frontend-design.md 的编写
  • Design Packet 的消费方式

十一、各类变更你怎么走

你要做什么 change_class 要 Cap-Spec? 要 HPRD? 要 QA verdict?
新功能 / 用户可见能力 capability ✅ ✅ ✅
修 Bug bugfix ❌ ❌ ✅（不可免除）
重构 / 性能优化 tech-improvement ❌ ❌ 仅影响用户行为时
改 kernel / gateway governance ❌ ❌ 创始人必须审批

Bugfix 防滥用：不得引入新 API / reason_code，单个 PR 不超过 500 行。

技术改动防滥用：不得包含业务逻辑变更；涉及 TECH-STACK-SPEC 锁定项则升级为 governance。

十二、常见问题

Q：AI 生成的代码，我需要 100% 理解吗？ A：你需要理解它做了什么、为什么这么做、是否符合设计意图。你不需要逐行背诵，但你需要能判断"这段代码是对的"。你是裁决者，不是打字员。

Q：pm-hprd-pass 一直不过怎么办？ A：找 PM 确认你的理解哪里偏了。HPRD 是双向沟通工具——如果连续不过，可能是 Cap-Spec 本身不够清晰，可以要求 PM 补充。

Q：我能直接 merge 自己的 PR 吗？ A：不能。所有 PR 需要 code owner review + CI 门禁 + qa-verdict + pm-acceptance（能力包流）。四权分离意味着你不能放行自己的代码。

Q：L4 AI 审计给了 warning 但不是 BLOCK，我需要处理吗？ A：L4 是 ADVISORY。P0 安全问题会升级为 BLOCK，其他的参考处理。建议认真看——AI 审计经常能发现人类遗漏的问题。

Q：后端和前端 PR 在同一个能力包里能一起提吗？ A：分开提。后端 PR 在 hl-platform，前端 PR 在 hl-app，各自走各自的门禁和 CODEOWNERS。但它们共享同一个 Cap-Spec 和 pm-acceptance。
