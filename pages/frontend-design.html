🎨 前端与设计团队指南

唤龙平台 AI 驱动协作模型
前端 5 人 + 设计师 2 人

适用角色：前端工程师（5 人，胡杰威 Lead）+ UI 设计师（2 人） 关联文档：全局工作流 · 产品团队指南 · 工程师团队指南 治理来源：TEAM-COLLABORATION-SPEC v2.1 §2.5 / §7.2 · UI-DELIVERY-SPEC v1.0

一、你们是谁

前端工程师 — UI Solution Owner

5 人团队，全部转 Flutter。使用两个仓库：hl-design-system（设计系统）和 hl-app（租户端多端应用）。hl-console-native 为创始人负责的 macOS 原生平台控制台，与前端团队无关。

胡杰威担任前端 Lead——不是因为 Flutter 经验最深，而是因为他在 AI 驱动上积极拥抱和实验，最适合带领团队完成转型。

在 AI-first 模型下，你的工作方式：

  • AI 生成 Flutter 代码为主，你负责意图定义、审查和裁决
  • 人工可补充，但所有 diff 统一经过 CI 门禁
  • 你需要读懂 Dart/Flutter 代码、定义组件接口和状态、理解 Flutter 特性（Widget/State/Provider 等）

UI 设计师 — 视觉/交互规范 Owner

2 人横向支持所有有 UI 界面的能力包。同时参与两条线：

  • 能力包线：为每个能力包产出 Design Packet
  • Foundation 线：定义 Token、组件规范

二、双线交付模型

你们和后端最大的不同：你们有两条生产线，而不是一条。

线 1：UI Foundation Lane（横向基础设施）

独立于能力包流程，属于 tech-improvement 类变更。这是设计系统的建设线。

阶段 谁做 产出 通过标准
1. Token 定义 设计师 DTCG-compatible token 文件 命名、层级、主题策略明确
2. Token 编译 前端负责人 Flutter ThemeData / ThemeExtension 可编译通过
3. 组件实现 前端 基础组件 + 状态规范 API、变体、状态齐全
4. Catalog / 文档 前端 + 设计 组件隔离工作台 use cases 适用状态齐全
5. 质量门 QA golden / a11y / smoke regression 和 a11y 通过
6. MCP/Skill 封装 前端 + AI 工程 AI 可消费的设计系统 T1.5 推进，非 T1 阻塞

关键：第 6 阶段（MCP/Skill 封装）在 T1.5 推进。T1 优先确保 Token → 组件 → Catalog 链路打通。

线 2：Capability UI Delivery Lane（能力包 UI 交付）

嵌入能力包五步流程，前端有自己的前置门和节奏，不再被后端吞掉。

阶段 谁做 输出 Gate
1. Cap-Spec Freeze PM Cap-Spec + 验收场景集 cap-spec-frozen
2. Design Freeze 设计师 design-packet.yaml 冻结入仓 design-ready
3. API/Mock Freeze 后端 / 契约 owner OpenAPI / mock / contract ref api-mock-ready
4. UI Input Ready 系统 设计 + contract 双输入到位 ui-input-ready = design-ready AND api-mock-ready
5. Frontend Design 前端工程师 frontend-design.md 前端互审
6. Foundation 分流 前端负责人 是否先起 Foundation PR foundation-change-required
7. 实现 前端 代码 + 组件隔离态 + 测试 Draft PR
8. UI 门禁 CI + QA + 设计/PM regression / a11y / conformance 按 UI-R 级别
9. PM 产品验收 PM 产品验收 pm-acceptance
10. 合并/发布 系统/运维 merge / release merge queue

硬规则：No UI Input Ready, No Frontend Coding

前端的真实输入不是一个（设计），而是两个（设计 + API contract/mock）。两个都就绪了才能开工。前端不该等"后端代码 ready"，而应等"API contract/mock ready"，在隔离态并行开发。

三、Foundation 与 Capability 的依赖握手

在能力包开工前，必须判断是否需要先改设计系统：

情况 处理
只复用现有 token + 现有组件 直接走 Capability UI Delivery Lane
需要新公共组件 先起 Foundation 变更，能力包引用 foundation version
需要新 token / 新主题语义 先起 Foundation 变更
只是能力包私有页面拼装 不得偷偷沉淀为"公共组件"

在 design-packet.yaml 中通过以下两个字段表达依赖：

  • foundation_change_required: true / false
  • foundation_version_ref: 依赖哪一版设计系统

四、Design Packet 详解（设计师必读）

Design Packet 是你的核心交付物。它不是 Figma 链接——它是仓库内的机读 YAML 文件。

存放位置：hl-app/docs/design-packets/{能力包}/design-packet.yaml

12 个必填字段

字段 你填什么
capability_id 对应哪个能力包（如 biz.flow.sales）
figma_refs 你的 Figma 设计稿 URL 列表
screen_ids 这个能力包涉及哪些屏幕/页面
state_matrix 每个页面要覆盖哪些状态（default/loading/error/empty 等）
token_version 绑定哪一版 token（如 v1.2.0）
component_refs 引用哪些基础组件（如 hl-button, hl-card）
breakpoints 响应式规则或不同设备尺寸
a11y_notes 无障碍约束（对比度、标签、触摸目标）
api_contract_ref 依赖的 API 或 mock 版本
foundation_version_ref 依赖哪一版设计系统
foundation_change_required 是否需要先改设计系统
frozen_at + owner 冻结时间 + 你的名字

版本钉死：冻结后前端按这个版本实现。要改就要解冻 → 修改 → 重新冻结。

五、UI 变更风险分层

不同变更走不同门禁，小修不被重流程拖死。

风险级别 示例 门禁要求
UI-R0 文案、小样式、现有组件内的小布局 design-review-note（Advisory，不阻塞）
UI-R1 新页面变体、现有组件组合的新界面 design-ready + ui-input-ready
UI-R2 新公共组件、新状态模式、大交互改动 R1 全部 + widget-catalog-pass + design-conformance-pass
UI-R3 全局主题/token、导航骨架、关键商业路径页面 R2 全部 + 创始人/前端负责人/PM 升级审查

PR 提交时在分类中声明 ui_risk_level。不确定就按 UI-R2。

六、前端的 UI 门禁体系

Check 什么时候触发 通过标准 R0 R1 R2 R3
design-ready 触碰 UI 变更 design-packet.yaml 存在且完整 — ✅ ✅ ✅
api-mock-ready 需要后端数据 API contract/mock 可消费 — ✅ ✅ ✅
ui-input-ready 前端开工前 以上两个都通过 — ✅ ✅ ✅
token-sync-pass 触碰 UI 样式 禁止硬编码颜色/间距 — ✅ ✅ ✅
widget-catalog-pass 新增公共组件 组件隔离工作台状态齐全 — — ✅ ✅
design-conformance-pass 新页面/大交互改动 实现与 Design Packet 对齐 — — ✅ ✅
ui-regression-pass UI 变更 golden/visual diff 通过 — ✅ ✅ ✅
ui-a11y-pass 触碰交互/布局 contrast/labels/target size — ✅ ✅ ✅
design-review-note 小修小补 设计 reviewer 留意见 ✅ — — —

八、Web → Flutter 转型说明

4 名 web 前端全部转 Flutter。在 AI-first 模式下：

  • AI 生成 Flutter/Dart 代码，你负责定义组件接口、状态结构、路由规划
  • 你需要读懂 Dart 语法、理解 Flutter Widget 树、Provider/Riverpod 状态管理
  • Web 前端对组件化、状态管理、响应式布局的理解可以直接复用
  • 通过实际能力包开发在 AI 编码过程中自然习得 Flutter

Figma 约束：Figma（当前工具选型）Code Connect 当前不支持 Flutter。所以：

  • Figma MCP 作为上下文输入（AI 读取设计信息），不作为权威绑定
  • Token 通过 DTCG 标准格式定义，使用 token 编译工具生成 Flutter ThemeData
  • 组件通过组件隔离工作台（如 Widgetbook（当前工具选型））进行隔离展示和测试

九、四种最大失败模式（请警惕）

  1. 设计没冻结，前端先写 → 返工地狱
  2. API/mock 没冻结，前端空等 → 产能浪费
  3. 新组件不走 Foundation，页面私造组件 → 设计系统碎片化
  4. 视觉回归过了，但设计一致性没守住 → 用户体验不统一

十、常见问题

Q：前端可以不等后端代码 ready 就开始吗？ A：可以，但你要等的是 API contract/mock ready，不是后端代码 ready。前端在隔离态并行开发 UI，后端提供 mock 接口即可。

Q：设计师的 Figma 链接还有用吗？ A：有用。Figma 链接放在 design-packet.yaml 的 figma_refs 字段里，作为设计来源定位。但 SSOT 是 design-packet.yaml，不是 Figma 链接。

Q：什么时候需要先改设计系统？ A：当你需要新 token（新颜色、新间距语义）、新公共组件（新按钮变体、新卡片类型）、或新交互模式时，先起 Foundation PR。在 design-packet.yaml 中标 foundation_change_required: true。

Q：UI-R0 的小修需要写 Design Packet 吗？ A：不需要。UI-R0 只需要 design-review-note（Advisory），设计 reviewer 留个意见就行，不阻塞合并。

Q：组件隔离工作台是什么？ A：类似 Storybook 的 Flutter 版工具（如 Widgetbook）。你在里面展示每个公共组件的所有状态（default/loading/error/empty/disabled），用于 QA 验收和设计师确认。
