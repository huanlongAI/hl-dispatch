🔺 Section 1: 四层门禁金字塔

唤龙平台采用从下往上递进式的质量把控体系。每一层的失败都会阻止 PR 合并。点击任何层级查看具体检查项。

📋 各层详细说明

🟨 L1 格式门禁 — 最快、最廉价的检查

特征：静态分析，无需运行代码，秒级反馈 · 判定：违反 = BLOCKING，不能合并

ktlint
Kotlin 代码风格检查（hl-platform）
dart format
Dart 代码格式检查（hl-app · hl-design-system）
yamllint
YAML 配置文件验证（hl-contracts · hl-factory）
markdown-lint
Markdown 文档规范（所有仓库）

🟨 L2 确定性门禁 — 规则明确、无歧义

特征：规则驱动，结果唯一确定，分钟级反馈 · 判定：违反 = BLOCKING，不能合并

Contract Schema
Cap-Spec 与 API 规范必须符合 JSON Schema
reason_code 检查
所有 error_reason 必须在 hl-contracts/reasoncodes/ 中定义
验收场景覆盖
capability_id 的验收场景覆盖率 ≥ 60%（G-026 规则）
token-sync
hl-app 中使用的设计令牌版本必须与 hl-design-system 一致
design-ready
UI 相关 PR 必须包含对应的 Design Packet（hl-app 专用）

🟢 L3 行为门禁 — 代码执行验证

特征：需运行代码的验证，分钟~小时级，提供客观证据 · 判定：失败 = BLOCKING，不能合并

unit test
单元测试 100% 通过。覆盖率 ≥ 80%（对关键路径要求 100%）
integration test
集成测试覆盖模块间接口。hl-platform 必须测试 API contract
golden test
hl-app 的 widget tree 变更需要 golden 测试（视觉回归检查）
a11y test
无障碍测试（屏幕阅读器、键盘导航）· hl-app 必须 WCAG 2.1 Level AA
acceptance test
QA 驱动的验收测试。必须覆盖 Cap-Spec 中定义的所有验收场景

🟣 L4 语义门禁 — AI 驱动的智能审计

特征：非阻断性（ADVISORY），为创始人和审阅者提供参考 · 判定：建议参考，不强制

AI 代码审查
哨兵自动扫描代码质量、安全风险、架构一致性
复杂度分析
检查 cyclomatic complexity、认知复杂度，建议重构
安全扫描
依赖漏洞、敏感信息泄露（API key、密码等）
架构守护
检查是否违反模块边界、是否引入不允许的跨层依赖

🛡️ Section 2: 哨兵（Sentinel）— CI 中的 AI 审查节点

🛡️

哨兵（Sentinel）是什么？

定位：CI 管线中的智能审查节点，运行在每个 PR 上，自动进行 AI 驱动的代码审计。

职责：

  • 契约一致性检查 — PR 的代码变更是否与 Cap-Spec 对齐，是否引入了超出设计范围的改动
  • 安全扫描 — 依赖漏洞、敏感信息泄露（如数据库密码、API key 暴露）、SQL 注入风险
  • 架构守护 — 是否违反了模块间的边界、是否引入了不允许的跨层依赖、是否破坏了分层隔离
  • 代码质量 — 圈复杂度、重复率、AI 生成代码的可维护性、代码异味检测

与门禁体系的关系

哨兵的检查结果注入到 L4 语义门禁层（ADVISORY），不直接阻断 PR，但为创始人和审阅者提供重要的决策参考。
创始人或工程师在看到哨兵的建议后，可以选择：

  • 接受建议，优化代码
  • 理解后明智地忽略（若有充分理由）

典型检查项

检查项 触发条件 建议行动
契约漂移 Cap-Spec 版本与代码不一致，或发现未授权的 API 签名变更 调整代码以符合 Cap-Spec，或更新 Cap-Spec 后获批
SQL 注入风险 检测到动态拼接 SQL 而非参数化查询 立即修复，使用 ORM 或参数化查询
秘钥泄露 检测到硬编码的 API key、密码、token 立即从代码中移除，使用环境变量或密钥管理服务
模块越界 hl-app 直接导入内部 API 而非通过 service layer 遵循分层架构，通过 service layer 中介
复杂度过高 圈复杂度 > 10 或认知复杂度 > 15 考虑重构成更小的函数或使用策略模式

实际工作流

  1. 工程师推送 PR
  2. CI 管线开始运行（L1 → L2 → L3）
  3. L4 并行运行哨兵检查，生成审查报告
  4. 哨兵报告作为评论出现在 PR 中（不阻断，仅供参考）
  5. 工程师查看报告，可选择优化代码
  6. 人工审阅（代码所有者 → QA → PM → 创始人）
  7. 通过审批后可合并

🤖 Section 3: 技术合伙人智能体（Technical Co-founder AI）

🤖

技术合伙人智能体是什么？

定位：创始人的 AI 决策辅助工具，不在 CI 管线中，而是在治理决策链中。

工作场景：创始人需要审阅涉及以下内容的 PR 时：

  • contract_touch — 涉及契约变更的 PR（新增/修改 reason_code、API 规范等）
  • founder_required — 架构决策、技术方案变更、治理规范调整

核心职责

  • 影响分析 — 此变更会影响哪些系统、哪些团队、哪些已上线的能力
  • 一致性检查 — 治理规范变更是否与现有的四权分离模型一致
  • 向后兼容性评估 — 契约变更是否会破坏现有的客户端实现
  • 风险识别 — 标出可能的风险点和需要特别关注的地方
  • 建议方案 — 如果存在问题，提出替代方案或改进建议

与门禁的区别

哨兵（Sentinel）：自动运行，给所有 PR 反馈，是 L4 层的一部分
技术合伙人智能体：手动触发，只在创始人审批时生成建议，是创始人的私密顾问

工作模式

  1. 创始人打开需要审批的 PR（通常标记了 founder_required 标签）
  2. 创始人点击"请求智能体分析"按钮
  3. 技术合伙人智能体快速生成：
    • 核心变更摘要
    • 影响范围分析
    • 风险评估
    • 建议的审批清单
  4. 创始人采纳或忽略建议，做出最终决策
  5. PR 审批完成

示例场景

场景：PM 提交 PR，在 hl-contracts/reasoncodes/ 中新增了 3 个 reason_code，涉及订单退货流程。

技术合伙人智能体的分析：
  • 新增的 3 个 reason_code 会被以下系统使用：hl-platform (订单服务) · hl-app (租户端) · hl-console-native (创始人后台)
  • 这是一个后向兼容的变更，不会破坏现有客户端
  • 建议：创建对应的验收场景文档，PM 需补充测试用例
  • 审批清单：PM 自己检查清单 ✓ · 工程师 review ✓ · QA 生成验收场景 ✓ · 可以审批

✅ Section 4: Required Status Checks — 组织级规则

以下三个 Required Checks 是 GitHub 组织级别的强制规则，出现在所有 PR 的"Merge button"旁边。必须全部通过才能合并。

Check Name 触发仓库 含义 审批权限
qa-verdict BLOCKING hl-platform, hl-app QA 团队必须在 PR 中显式批准（Approve review），验证功能符合验收场景 @hl-qa-team
pm-acceptance BLOCKING hl-platform, hl-app PM 必须在 PR 中显式批准，确认代码实现与 Cap-Spec 一致 @hl-pm-team
pm-hprd-pass ADVISORY 所有 流程内检查：验证 HPRD（工程师理解确认文档）是否已创建。非组织级 Ruleset，仅供参考 @hl-pm-team

⚠️ 重要区别：
qa-verdict 和 pm-acceptance 是 GitHub 组织级 Ruleset，强制所有 PR 通过。
pm-hprd-pass 是流程内检查（在 hl-factory 中定义），不是组织级，仅提示信息。

📌 核心要点

  • 四层递进：L1（快速）→ L2（规则）→ L3（实现）→ L4（智能），从廉价到昂贵，从机械到智能
  • 哨兵驱动：🛡️ Sentinel 在 L4 层自动运行，为所有 PR 提供 AI 审计反馈
  • 治理决策：🤖 技术合伙人智能体不在 CI 中，而是在创始人审批时作为决策助手
  • Required Checks：qa-verdict · pm-acceptance 是强制检查点，任何 PR 都不能绕过
  • 人机协作：自动化做好 L1~L4，人类做好人工审阅和最终决策
