GitHub + 云效 流水线指南

PR（Pull Request，合并请求）从提交到上线，在 GitHub 与阿里云云效之间如何流转 · 谁按哪个按钮 · 在哪一关会卡住。

可视化 + 规范 · 二合一 DRAFT · 2026-04-22 首发

§1 · 一分钟读懂

核心两句话

① GitHub 负责代码合入：PR → Required Status Checks（必须通过的强制检查）→ Merge（合并）。
② 云效 负责制品与部署：合入 main 后触发镜像构建 → 部署编排（蓝绿/灰度）→ 健康巡检。

这两套系统职责不重叠：GitHub 是代码合入门禁的权威，云效是运行态部署的权威。跨系统的衔接点只有一个——合入 main 的那一刻，GitHub Webhook 通知云效开始构建。

为什么需要两套：GitHub 的 Actions 做不了 蓝绿/金丝雀发布（Blue-Green / Canary，渐进式上线）、回滚、多可用区编排；云效做不了代码评审与 Required Status Checks（组织级强制检查）。分工合作，谁专业谁来做。

§3 · GitHub 侧 · 代码合入

GitHub 这一侧是门禁方，核心产物有三类：分支保护（Branch Protection）、GitHub Actions（CI 脚本）、Required Status Checks（组织级强制检查）。

§3.1 · 分支策略

分支允许谁推送保护规则用途
main 禁止直推 PR only · 强制 Required Checks · 禁止 force push 发布基线 · 与云效 prod 对齐
feature/* 工程师 无保护 · 建议每次 rebase main 日常开发
release/* Ops Lead（曾正龙） 只允许 cherry-pick 合入 main 的 commit 稳定版回归测试 · 补丁发布

§3.2 · Required Status Checks（组织级强制检查）

⚠️ 权威源在 门禁体系详解 §4

本页不重复 Required Status Checks 的定义与审批权限表。以下只列出在流水线上的落位：哪个检查出现在 PR 的哪个阶段。

Check来源触发时机典型耗时
qa-verdict BLOCKING QA 团队后台（人工） PR 打开后 · QA 完成验收场景测试 数小时 – 1 天
pm-acceptance BLOCKING PM 在 PR 页面点击 Approve 代码 review 与 QA 验收完成后 约 30min
pm-hprd-pass ADVISORY hl-factory 流程内检查（自动） PR 创建时立即触发 约 10s
GitHub Actions CI .github/workflows/*.yml 每次 push 到 PR 分支 约 2–8min（L1–L4 完整跑）

§3.3 · Merge 策略

三种合并方式的选择规则：

  • Squash and merge（压缩合并，默认）—— 适用于 99% 的能力包 PR。把 feature 分支的 N 次 commit 压成一个 commit 进 main，保持主干历史干净。
  • Rebase and merge（变基合并）—— 仅限 Ops Lead 用于 release/* 分支 cherry-pick 场景。
  • Merge commit（合并提交） —— 已禁用。避免主干出现 "Merge pull request #xxx" 噪声。

§3.4 · Merge Queue（合并队列）

当多个 PR 同时就绪时，GitHub Merge Queue 串行化合并：每个 PR 在进入 main 前都以最新 main 为 base 重跑一遍 CI。这保证语义合并——两个单独绿灯但合并后冲突的 PR 不会同时进主干。

§4 · 云效 侧 · 制品与部署

云效接管 main 合入之后的所有工作。分两个大块：镜像构建和部署编排。

§4.1 · 镜像构建（Container Build）

1

Webhook 触发

GitHub main 合入 → 云效流水线收到 push 事件，拉取最新 commit。

2

Docker build

按 Dockerfile 构建镜像，打上 tag（格式：<branch>-<commit-sha7>-<timestamp>）。

3

镜像扫描

Trivy 扫高危漏洞 · License 合规检查（非 GPL-3.0）。任一失败阻断后续部署。

4

推送到制品仓库（ACR）

阿里云容器镜像服务 Container Registry。镜像保留期：prod 标签永久；非 prod 标签 30 天。

§4.2 · 部署编排（Deployment Pipeline）

四段递进 · 全程可观测 · 任一段失败可一键回滚：

A

staging 部署（自动）

合入 main 后镜像自动推 staging 环境。QA 在 staging 做最终冒烟测试（约 30min）。

自动QA 验证
B

canary 部署（金丝雀，5% 流量）

Ops Lead（曾正龙）在云效按钮面板点击"进入 canary"。5% 流量切到新版本，观察 30min。

曾正龙 按钮
C

prod 全量（100% 流量）

canary 指标绿灯（错误率 < 0.5% · p99 延迟无退化） → Ops Lead 点击"切 prod"。老版本保留 24h 作回滚兜底。

曾正龙 按钮
D

稳定期 + Released

prod 全量 24h 无回滚 → 能力包状态自动回写为 Released。

自动

§4.3 · 回滚（Rollback）

🚨 回滚只有一个按钮 · 只有 Ops Lead 能按

触发条件：canary/prod 指标告警（错误率 > 1% · p99 退化 > 30% · 健康巡检失败）。云效控制台一键切回上一个 prod 标签，无需代码 revert。回滚后 24h 内 Ops Lead 必须提交 事故复盘。

§5 · 按钮权在流水线上的落位

流水线的关键决策点都由按钮权控制。按钮权 = 物理上能按下那个按钮的人。以下 5 个按钮贯穿整条流水线：

Merge button
INV-0 童正辉
GitHub PR 页面 · 所有 Required Checks 绿灯后可按
qa-verdict
李云峰（QA Lead）
QA 团队后台 · 验收场景全通过后打 Approve
pm-acceptance
能力包 PM Owner
GitHub PR 页面 · Approve review（团队统筹由 PM Lead 邹骢负责）
pm-hprd-pass
能力包 PM Owner
hl-factory 流程自动检 · PM 在 Cap-Spec 阶段完成 HPRD 后自动绿灯
进 canary / 切 prod / 回滚
曾正龙（Ops Lead）
云效控制台按钮面板 · 所有环境切换与回滚
📘 按钮权与签字边界的区别

按钮权在物理层（谁能点），签字边界在治理层（谁要为决定负责）。两者经常重合但不等价——例如 INV-0 按 Merge button 不代表 INV-0 要为代码质量负责，QA 的 qa-verdict 才是质量签字。完整签字边界见 团队代表 · 签字矩阵。

§6 · 失败路径与熔断

流水线不是"一路绿灯直到上线"的理想路径，60% 的 PR 都会在某个节点卡住。以下是三类常见失败及处置：

§6.1 · GitHub 侧失败

  • CI 红灯（L1 格式 / L2 构建 / L3 单测 / L4 契约失败）—— 工程师修，push 新 commit，Actions 自动重跑。
  • qa-verdict 打回（Request Changes）—— QA 在 PR 里写明拒绝原因，工程师修，QA 复测。
  • pm-acceptance 打回—— PM 认为实现与 Cap-Spec 不一致，工程师与 PM 对齐，必要时修改 Cap-Spec 走治理流程。
  • Merge Queue 冲突—— 前车 PR 改动与本 PR 冲突，Queue 自动把本 PR 踢出，工程师 rebase 后重新入队。

§6.2 · 云效构建失败

  • Docker build 失败—— Dockerfile 与 main 代码不同步。工程师补 PR 修 Dockerfile。
  • 镜像扫描阻断（Trivy 高危）—— 升级依赖，补 PR。紧急情况走 WAIVER（临时豁免）走治理审批。

§6.3 · 云效部署失败（熔断）

🛑 熔断阈值（自动回滚触发）
  • canary 阶段：错误率 > 1% 持续 5min
  • prod 切量：p99 延迟退化 > 50%
  • 健康巡检：任意核心接口 5xx > 10 次 / min

以上任一达成 → 云效自动停止切量 + 告警 Ops Lead。由 Ops Lead 决定手动回滚 or 继续观察。
