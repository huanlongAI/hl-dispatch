GitHub + 云效 流水线指南

唤龙平台的流水线是一条 "上游 → 中段 → 下游" 的单向河：GitHub 上游承载契约与知识源（创始人维护），阿里云云效承载中段工作区（工程师在云效创建流水线 + 发起云效 PR + CI 验证 + 部署编排），验证完成后回合 GitHub 下游作为最终基线。本页把这条完整路径画成交互式流程图，并用二级 Tab 组织所有细节。

12 步交互演示 上游契约源 · 中段云效 PR · 下游最终合入 按钮权落位 · 熔断规则

§1 · 一分钟读懂

流向一句话

GitHub 上游（契约 + HK） → PM 拉取设计 Cap-Spec → 工程师接收 → 工程师产出 HPRD / 设计 / 测试规格 → 云效接入 GitHub 拉取 → 工程师在云效创建流水线 + 发起云效 PR → HPRD 内部分发给 PM/FE/Design/QA 审计 → 云效 CI + 部署验证 → 回合 GitHub 下游 main（INV-0 最终合入）。

三段式模型

契约源（hl-contracts）+ HK（Human Knowledge）仓库
创始人 / INV-0 维护

→

② 中段 云效

⚠️ 与其他页面可能有张力

本页把 Required Status Checks（qa-verdict / pm-acceptance / pm-hprd-pass）定位在 云效 PR 上。 gate-levels.html §4 当前把它们描述为 "GitHub 组织级规则"——这一点与纠正后的流水线模型存在定位差异。 治理层以 hl-contracts/TEAM-COLLABORATION-SPEC v2.1.3-patch 为准；站点层待一致化更新。

为什么分三段

  • 上游 GitHub = 契约主权：契约（Contract）与 HK（Human Knowledge）是语义主权之所在，创始人独家维护，所有下游必须引用这个源。
  • 中段 云效 = 工作区：阿里云云效提供完整的工程师协作面：代码同步、PR 评审、CI 编排、制品构建、蓝绿 / 金丝雀部署、按钮化发布控制。工程师 90% 时间待在这里。
  • 下游 GitHub = 最终基线：云效 PR 一系列（CI + 审计 + 部署验证）全通过后，才由 INV-0 把代码回合到 GitHub main，作为下一轮能力包的最新契约基准。

§3 · GitHub 侧 · 上游契约源 + 下游最终合入

GitHub 在唤龙流水线里扮演 "首尾两端"：流程从 GitHub 上游契约开始，绕一圈云效后回到 GitHub 下游 main。

§3.1 · 上游契约源

GitHub 上游承载两类权威资产，全部由创始人 / INV-0 维护：

仓库内容写权限下游谁拉取
hl-contractsTEAM-COLLAB-SPEC · SITE-PUB · RULINGS · SIGN-OFFINV-0 童正辉PM（设计 Cap-Spec 参照）· 工程师（HPRD 参照）· QA（测试规格书参照）
HK（Human Knowledge 仓库）业务域知识库 · 领域模型 · 已裁决先例INV-0 童正辉PM / 工程师 / 设计
hl-platform / hl-app 等代码仓库实际代码PR only（下游合入）云效自动拉取

§3.2 · 下游最终合入

云效 PR 完成 "一系列"（CI + Required Checks + 部署验证）之后，代码才回合到 GitHub main。这是 INV-0 按 Merge button 的最后一步。

分支允许谁推送保护规则用途
main禁止直推（仅云效 PR 合入）云效"一系列"全绿后 INV-0 按 Merge发布基线 · 下一轮能力包契约起点
release/*Ops Lead（曾正龙）cherry-pick补丁专用稳定版回归 · hotfix

§4 · 云效 侧 · 中段工作区

云效是工程师 90% 时间所在的地方。从"接入 GitHub 拉取"开始，到"部署验证全绿"结束，整个中段都在云效。

§4.1 · 云效接入 GitHub（拉取上游）

1

Webhook 绑定

云效项目绑定 GitHub 上游仓库，任何 push / PR 事件实时同步。

2

镜像 clone

云效在自己的仓库镜像里维护一份 GitHub 上游的同步副本，工程师在云效侧做所有开发操作。

3

凭证托管

云效使用 OAuth App 或 GitHub App 方式访问 GitHub 上游，凭证由 INV-0 设置，工程师不接触。

§4.2 · 工程师创建云效流水线

工程师接收 Cap-Spec + 产出 HPRD/设计/测试规格后，在云效控制台创建流水线：

A

定义 stage

格式检查 → 构建 → 单测 → 契约测试 → 镜像构建 → staging 部署 → canary → prod。

B

绑定审批节点

qa-verdict / pm-acceptance 两个 BLOCKING 审批节点挂在云效 PR 上，由对应 Lead 按。

C

本地开发 + commit

工程师基于云效 clone 的仓库做 feature 分支开发，commit 推回云效侧。

§4.3 · HPRD 内部分发与审计

工程师把 HPRD 打包分发给四方（PM / 前端 / 设计 / QA），各方审计：

  • PM 审：HPRD 与 Cap-Spec 是否一致（语义主权对齐）
  • 前端审：HPRD 的交互要求、组件契约是否可实现
  • 设计审：视觉 / 交互规范是否完整
  • QA 审：测试规格书可验收，HPRD 场景全覆盖

四方全部签字通过后，工程师才能发起云效 PR。

§4.4 · 云效 PR + "一系列"

云效 PR 阶段执行主体典型耗时BLOCKING?
① GitHub Actions L1–L4 CI（格式 / 构建 / 单测 / 契约）CI 自动2–8 minBLOCKING
② qa-verdict 审批李云峰 / QA Lead数小时 – 1 天BLOCKING
③ pm-acceptance 审批能力包 PM Owner约 30 minBLOCKING
④ pm-hprd-pass（自动）hl-factory 流程内检查10 sADVISORY
⑤ 部署验证：staging 自动 → canary 5% → prod 100%Ops Lead / 曾正龙1 – 数天BLOCKING

§4.5 · 部署验证细则

α

staging 部署（自动）

云效 PR 的 CI / 审批通过后自动推 staging 环境。QA 冒烟 ~30 min。

β

canary 5%（Ops Lead 按）

曾正龙在云效按钮面板点击"进 canary"。观察 30 min。

γ

prod 100%（Ops Lead 按）

canary 指标绿灯（错误率 < 0.5%）→ 曾正龙按"切 prod"。老版本保 24 h。

δ

回合 GitHub 下游

部署验证全过 → INV-0 按 GitHub Merge button → main 前进一个 commit → 能力包状态自动回写 Released。

§5 · 按钮权在流水线上的落位

流水线的关键决策点由 按钮权 控制。以下 5 个按钮贯穿整条流水线：

契约仓库写权
INV-0 童正辉
GitHub 上游 · hl-contracts / HK 写权 · 语义主权根
qa-verdict
李云峰（QA Lead）
云效 PR 审批 · 验收场景全通过后打 Approve
pm-acceptance
能力包 PM Owner
云效 PR 审批 · HPRD 与 Cap-Spec 一致性确认（PM Lead 邹骢 团队统筹）
pm-hprd-pass
能力包 PM Owner
hl-factory 自动检 · HPRD 存在即绿 · ADVISORY
进 canary / 切 prod / 回滚
曾正龙（Ops Lead）
云效控制台按钮面板 · staging/canary/prod 切换 · 一键回滚
GitHub Merge button
INV-0 童正辉
GitHub 下游 · 云效"一系列"全过后 · 合入 main
📘 按钮权 vs 签字边界

按钮权 = 物理层（谁能点）。签字边界 = 治理层（谁要为决定负责）。两者经常重合但不等价：INV-0 按 GitHub Merge 不代表 INV-0 要为代码质量负责，qa-verdict 的按钮（李云峰）才是质量签字。完整签字边界见 团队代表 · 签字矩阵。

§5.1 · 按钮时序

  1. INV-0（上游）更新 hl-contracts / HK → 契约源就绪
  2. PM Owner 拉取上游 + 出 Cap-Spec
  3. 工程师 接收 Cap-Spec + 产出 HPRD/设计/测试规格
  4. 云效 自动 拉取 GitHub 代码镜像
  5. 工程师 在云效创建流水线 + 开发 commit + 发起云效 PR
  6. PM+FE+Design+QA 四方审计通过
  7. pm-hprd-pass（自动）· 李云峰 打 qa-verdict · PM Owner 打 pm-acceptance
  8. 曾正龙 按 "进 canary" → "切 prod"（部署验证）
  9. INV-0 按 GitHub Merge button → 代码回合下游 main → Released

§6 · 失败路径与熔断

流水线不是"一路绿灯到上线"的理想路径，60% 的能力包都会在某个节点卡住。三类典型失败：

§6.1 · 中段审计打回（HPRD 四方审计）

  • PM 审 HPRD 与 Cap-Spec 不一致 → 工程师修 HPRD，重新分发。
  • 前端 / 设计 审觉交互规范缺失 → 工程师补齐设计 spec，重新分发。
  • QA 审测试规格书不完整 → 工程师补测试场景，重新分发。

四方审计打回不生成云效 PR，所以不触发 CI。工程师修改后重走分发 + 审计。

§6.2 · 云效 PR 内失败

  • GitHub Actions CI 红灯（L1–L4 任一失败）—— 工程师修 + push 新 commit，CI 自动重跑。
  • qa-verdict 打回（Request Changes）—— QA 在 PR 中写明拒绝原因 → 工程师修 + QA 复测。
  • pm-acceptance 打回 —— PM 认为实现与 Cap-Spec 不一致 → 工程师与 PM 对齐，必要时修 Cap-Spec 走治理流程。

§6.3 · 部署熔断

🛑 熔断阈值（自动停止切量 · 告警 Ops Lead）
  • canary 阶段：错误率 > 1% 持续 5 min
  • prod 切量：p99 延迟退化 > 50%
  • 健康巡检：任意核心接口 5xx > 10 次 / min

熔断发生 → Ops Lead 决定手动回滚（一键切回上个 prod 标签）或继续观察。回滚不需要代码 revert，只切流量。回滚 24 h 内 Ops Lead 提交事故复盘。

§6.4 · 在流程图里模拟
