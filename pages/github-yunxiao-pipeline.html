GitHub + 云效 双仓并行指南

唤龙平台的流水线是一条 "双仓并行" 的协同轨道：GitHub 是永远的主轨，承载契约、PR、合并门禁与审计；云效是并行执行器，从工程师接入起跑 CI、生成制品、部署到阿里云，通过 GitHub App 席位把结果回推为 GitHub PR 的 Status Check。代码与 HPRD 在 GitHub 主仓与云效副本之间双仓同步；INV-0 对云效黑盒，只看 GitHub PR 上的绿勾。

双仓并行模型 6 桩按钮权 GitHub App 席位机制

§1 · 一分钟读懂

模型一句话

GitHub 永远主轨承载 PR、合并门禁与审计；云效从工程师接入起并行执行 CI、生成制品、部署到阿里云；两者通过 GitHub App 席位 双向沟通，云效把 CI 结果回推为 GitHub PR 的 Status Check；INV-0 对云效黑盒，只看 GitHub PR 上的绿勾。

双仓并行模型

GitHub · 主轨（永远的合并门禁 + 审计平面）

① 上游契约层（纯 GitHub · 无云效）

  • hl-contracts · SSOT 契约 · INV-0 + PM 写权
  • team-memory · 团队 AI 记忆沉淀 · PM 主力场景

② 代码仓 PR 主战场

  • 代码仓：hl-platform · hl-framework · hl-console-native · hl-scene-app（规划）· hl-scene-design-system（规划）
  • PR 承载 Required Status Checks：L1-L4 CI（BLOCKING · 云效回推）· pm-hprd-pass（ADVISORY）· pm-acceptance（BLOCKING）· qa-verdict（BLOCKING）
  • INV-0 Merge button = 最终合并门禁
↕ GitHub App 席位（云效 bot 拉代码 / 回推 Status Check）

云效 · 并行执行器（工程师接入起 · INV-0 黑盒）

  • 双仓镜像：GitHub 主仓 ↔ 云效副本（App 同步）· 代码 + HPRD + 开发设计 + 测试规格全部同步
  • HPRD 走代码仓 /hprd/ 目录 · 受 Git 版本管理
  • 工程师在云效创建流水线 · 跑 L1-L4 CI · 生成制品 · 部署到阿里云（staging → canary → prod）
  • 通过 GitHub App 席位回推 GitHub PR Status Check
  • 曾正龙按云效按钮：staging / canary / prod / rollback

INV-0 不关心流水线细节 · 只看 GitHub PR 上的绿勾

与 gate-levels.html §4 一致

本页与 gate-levels.html §4 完全一致：Required Status Checks 定位在 GitHub PR（不是云效 PR）· L1-L3 CI 的执行载体是云效 · 结果通过 GitHub App 回推为 GitHub PR 的 Status Check。

为什么选双仓并行

  • GitHub 合并门禁统一可审计：所有 PR、Status Check、Review、Merge 事件都在 GitHub，审计平面单一，INV-0 和治理层永远看一个地方。
  • 云效执行环境灵活：CI 跑批、制品库、阿里云部署按钮化、蓝绿 / 金丝雀发布工具链，云效更贴合国内云环境，且不与 GitHub Actions 配额争抢。
  • INV-0 不必关心流水线细节：云效对 INV-0 完全黑盒，创始人语义主权留在 GitHub 端，执行细节由工程师 + 运维 Owner 在云效内部协作。
  • 工程师在云效工作，所有决定都在 GitHub PR 上留痕：开发 commit 推 GitHub → 云效 App 拉取 → 云效跑 CI → 回推 GitHub PR Status Check → PM / QA 在 GitHub PR 上 approve → INV-0 按 GitHub Merge。每一个决定点都有 GitHub PR 事件作为审计证据。

§5 · 云效 · 并行执行器（INV-0 黑盒）

云效的定位是 工程师接入后的并行执行器：Step 3 先把 HPRD / 开发设计 / 测试规格纳入 GitHub 主仓审计基线；从 Step 5 feature 分支开发起，云效副本才开始并行拉取代码、跑 CI、生成制品、部署到阿里云，并通过 GitHub App 席位把结果回推为 GitHub PR 的 Status Check。INV-0 对云效完全黑盒，只看 GitHub PR 上的绿勾。

§5.1 · GitHub App 席位机制

1

Webhook 事件路由

云效 App 在 GitHub 组织级安装，监听 push / PR open / PR update / PR merge 等事件，事件触发即拉取最新代码副本。

2

OAuth / GitHub App 凭证

云效以 GitHub App 身份访问 · 组织级权限含：仓库 read / PR write comments / Commit statuses write / Workflow read。凭证由 INV-0 一次性配置，运维（曾正龙）日常维护。

3

Status Check 回推

云效流水线跑完 L1-L4 CI 后，通过 GitHub Commit Status API 把结果（success / failure / pending）写回 GitHub PR 对应 commit，PR 页面 Required Checks 列表即时变色。

§5.2 · 工程师接入点（Step 5 云效并行起）

工程师的云效接入时点 = 能力包 Step 5（feature 分支开发），Step 3 只是 HPRD / 开发设计 / 测试规格进入 GitHub 主仓：

A

Step 3：HPRD 先入 GitHub 主仓

工程师在代码仓 /hprd/ 目录提交 HPRD · 开发设计 · 测试规格文档（全部 Git 版本管理）。此时先形成 GitHub 审计基线，不把它画成云效并行起点。

B

Step 5：云效创建流水线

工程师进入 feature 分支开发时，在云效控制台创建对应流水线：定义 stage · 绑定阿里云部署目标 · 配置熔断阈值。流水线本身是云效的资产，不同步到 GitHub。

C

Step 5：feature 分支开发 + 云效拉取

工程师 commit 推 GitHub feature 分支 · 云效 App 从这一刻开始拉取同步 · 云效侧触发 CI · CI 结果通过 Status Check 回推到 GitHub PR。

§5.4 · 制品 + 部署验证

α

γ

prod 100%（运维 Owner 按）

δ

§5.5 · INV-0 黑盒边界

INV-0 对云效黑盒

INV-0 不登录云效 · 不查看云效流水线日志 · 不直接按云效按钮。曾正龙（运维 Owner）的 B5 按钮由云效承载，与 GitHub 无关；INV-0 的 B4 GitHub Merge 永远前置于 B5 云效部署。流水线的治理视图对 INV-0 仅是一个东西：GitHub PR 上的 Required Status Checks 和 Merge button。

📘 按钮权 vs 签字边界

按钮权 = 物理层（谁能点）· 签字边界 = 治理层（谁要为决定负责）。两者经常重合但不等价：INV-0 按 GitHub Merge 不代表 INV-0 要为代码质量负责，qa-verdict 的按钮（李云峰）才是质量签字。完整签字边界见 团队代表 · 签字矩阵。

§7 · 失败路径与文档边界

§7.1 · 失败路径与熔断

流水线不是"一路绿灯到上线"的理想路径，60% 的能力包会在某个节点卡住。三类典型失败 + 演示按钮：

HPRD 审计打回

  • PM 审 HPRD 与 Cap-Spec 不一致 → 工程师修 HPRD，push 到代码仓 /hprd/ 目录 · 双仓同步 · 重新 request review。
  • QA 审测试规格书不完整 → 工程师补测试场景 · push · 同步。
  • 设计审视觉规范缺失 → 工程师补设计 spec · push · 同步。

HPRD 文档本身不直接跑 CI，但 HPRD 的 PR diff 会在 GitHub PR 上被 PM / QA / 设计通过 Review Comments + pm-hprd-pass ADVISORY 审计。打回 = Request Changes · 工程师修改后重走 Review。

GitHub PR Status Check 红灯

  • L1-L3 CI 红（云效回推 failure）· 工程师修 + push 新 commit · 云效重跑 · 回推新 Status Check。
  • qa-verdict Request Changes（李云峰在 GitHub PR 上打回）· 工程师修 + QA 复测 + 重新 approve。
  • pm-acceptance Request Changes（PM Owner 认为实现与 Cap-Spec 不一致）· 工程师与 PM 对齐 · 必要时修 Cap-Spec 走 B1 治理流程。

部署熔断（云效侧）

🛑 熔断阈值（云效自动停止切量 · 告警运维 Owner）
  • canary 阶段：错误率 > 1% 持续 5 min
  • prod 切量：p99 延迟退化 > 50%
  • 健康巡检：任意核心接口 5xx > 10 次 / min

熔断发生 → 曾正龙决定手动回滚（B5 · 一键切回上个 prod 标签）或继续观察。回滚不影响 GitHub PR 合入状态：代码已在 main · 仅回退流量。24 h 内运维 Owner 提事故复盘写回 team-memory。
