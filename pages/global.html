核心理念

🏛️ 规则即治理

创始人不审每个 PR，而是通过 CODEOWNERS、门禁规则、Required Status Checks 行使治理权。规则自动执行，人只在规则覆盖不到时介入。

🤖 AI-First 编码

默认由 AI 生成代码，工程师负责意图定义、审查和裁决。人工可补充，但所有 diff 统一经过 CI 门禁，不论来源。

🔐 四权分离

语义主权（PM + 创始人）、技术实现（工程师）、独立质量（QA）、组织治理（创始人）——同一条变更的四种 owner 不得完全重合。

📋 机读化验收

QA verdict 和 PM acceptance 不再是口头确认，而是 GitHub Required Status Checks。机器可验证，不可绕过。

四权分离详解

◆ 语义主权

Owner：创始人（全局契约 SSOT）+ PM（能力包语义 SSOT）
业务语义的真相源头——PM 定义能力包要做什么，创始人定义全局规则和契约边界。能力包语义不得越出契约包络。

◆ 技术实现

Owner：工程师
技术方案 + AI-first 编码 + 实现。PM 不审技术设计，工程师不定义业务需求。

◆ 独立质量

Owner：QA
验收测试 + 质量放行。工程师不能放行自己的代码。qa-verdict 是 Required Status Check。

◆ 组织治理

Owner：创始人
规则、门禁、契约核心、内核路径。执行者不能修改自己要遵守的规则。

🔄 能力包交付流程DRAFT

五步交付流（Capability Flow）——新功能、用户可见能力变化

1

Cap-Spec Freeze · 能力包规格冻结

PM 编写 Cap-Spec（规格书 + 验收场景集 + 业务码提案），UI 设计师补充视觉/交互设计。若涉及 reason_code 变更，须创始人先批（治理前置门）。PM 确认冻结后进入下一步。

PM 主导 设计师协同 创始人（如涉及契约）
2

Design + HPRD · 设计冻结与理解确认

工程师阅读 Cap-Spec，产出 HPRD（理解确认件）交 PM 审批；同时产出 design.md（技术设计）交工程师互审。QA 同步编写 acceptance-manifest.yaml。设计师产出 Design Packet 入仓。
硬规则：pm-hprd-pass 未通过前，不得开始实现。

工程师 主导 PM 审批 HPRD QA 编写 manifest 设计师 Design Packet
3

Implementation + Tests · AI-First 编码

工程师通过 AI-first 编码实现功能代码、单元测试、集成测试和可运行 demo。QA 同步编写验收测试套件。前端在 ui-input-ready（设计 + API mock 双就位）后开始实现。工程师提交 Draft PR。

工程师（AI 编码） QA 编写验收测试
4

Gates + Acceptance · 门禁验收（并行）

四条线并行：CI 门禁自动执行（L1-L3 BLOCKING），AI 审计（L4 ADVISORY），QA 独立验收出具 qa-verdict，工程师互审 Code Review。全部通过后 Draft PR 转 Ready for Review。

CI 自动门禁 L1-L3 QA 验收 verdict Code Review AI 审计 L4
5

PM Acceptance + Merge · 产品验收合并

PM 在可交互版本上执行产品验收（pm-acceptance status check），验收通过后进入 Merge Queue。如涉及 founder-owned 路径，创始人额外审批。

PM 产品验收 创始人（如涉及核心路径）
6

Post-Merge: Release · 发布上线

运维工程师按发布策略部署。staging 环境部署后 QA 回归验证，通过后推 prod。运维监控 + 回滚预案就位。

运维执行 QA 回归验证

🐛 Bugfix 流详情

▼

适用于已上线功能的缺陷修复，不引入新业务语义。

  • P0 紧急：直接修复 → 最小 review（1 人）→ CI 全绿 → qa-verdict → merge → 立即发布 → 事后补 Root Cause
  • P1-P3 常规：Draft PR → CI 全绿 → qa-verdict → code owner review → merge → 跟随常规发布
  • 防滥用：不得引入新 API / reason_code；单个 PR 不超过 500 行（不含测试）

🔧 技术改动流详情

▼

不影响业务语义的技术改进：依赖升级、重构、性能优化、CI 配置。

  • 不需要 Cap-Spec / HPRD / PM 验收
  • 触碰 kernel / gateway / framework → founder_required = true
  • CI L1-L3 必须全绿
  • 防滥用：不得包含业务逻辑变更；涉及 TECH-STACK-SPEC 锁定项 → 升级为 governance

🏛️ 治理/内核流详情

▼

对契约核心、门禁规格、CODEOWNERS、内核架构的变更。

  • 创始人全程审批（必须）
  • CI L1-L3 必须全绿
  • 若影响在进行中的能力包 → 通知相关 PM + 工程师
  • 治理变更立即生效，不等发布窗口
🚦 门禁体系DRAFT → 深入详解：哨兵 · 技术合伙人 · L1-L4

四层门禁 · 自动执行 · 不可绕过

L1

格式性门禁 — BLOCKING

编译通过、类型检查、代码格式（ktfmt）、License Header、YAML 语法

L2

确定性门禁 — BLOCKING

契约一致性、模块边界、ReasonCode 对齐、循环依赖、API 路径规范、G-026 验收覆盖率（manifest-based）、Spec 漂移检测

L3

行为性门禁 — BLOCKING

单元测试、集成测试（Testcontainers）、G-023 验收场景回放、DoD 静态验证、回归集

L4

语义性审计 — ADVISORY

双模型交叉审计（GPT 初审 + Claude 仲裁）、意图一致性、代码可读性、安全审计（P0 安全问题升级为 BLOCK）

📋 PR 分类字段（每个 PR 必填）

▼

在 PR body 中声明以下 YAML 块：

change_class: capability    # capability | bugfix | tech-improvement | governance
contract_touch: false       # true | false
founder_required: false     # true | false

自动升级规则：

  • contract_touch = true → founder_required 必须 true
  • change_class = governance → founder_required 必须 true
  • change_class = capability 且 contract_touch = false → CI 自动放行
