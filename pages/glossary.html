—— 核心治理概念 ——

契约Contract / SAAC核心概念
hl-contracts 仓库里由创始人维护的全局规则集合——包括数据结构、接口、命名、权限、分层约束等。所有能力包都必须在契约允许的范围内实现。契约是"宪法"，能力包是"立法"。
别名：SAAC（Semantic Architecture As Contract）｜全局契约
契约边界Contract Boundary核心概念
契约所圈定的能力包可活动范围。能力包可以在边界内自由定义业务语义，但不能越界（比如私自改表结构、跨过网关）。早期文档中曾叫"契约包络（envelope）"，是同一个意思，已统一为"契约边界"。
曾用词：契约包络、SAAC 包络
能力包Capability Package核心概念
一个可独立交付的最小业务单元——类似"微服务 + 它的前端 + 它的契约片段 + 它的验收场景"的打包。PM 定义能力包"要做什么"，工程师实现"怎么做"。
对比：传统项目里的"模块/Feature"
语义主权Semantic Sovereignty核心概念
"业务到底是什么意思"的最终解释权。创始人对全局契约有语义主权，PM 对单个能力包内部有语义主权。工程师不能私自改变业务含义。
通俗说法：谁说了算
四权分离Power Separation核心概念
语义主权（创始人+PM）、技术实现权（工程师）、独立质量权（QA）、组织治理权（创始人）。四权互相制衡，任何一权都不能单独通过变更。
详见：四权分离详解
技术合伙人智能体Technical Partner Agent核心概念
Claude/Codex 等 AI 助手在本组织的定位——不是工具而是"合伙人级"的技术执行者，承担大部分编码、测试、运维脚本工作。人类负责提需求、审语义、把方向。
对应角色：AI-First 开发模式

—— 流程与工作流 ——

Cap-SpecCapability Spec流程
能力包的规格书（三段：业务目标 / 验收场景 / 契约影响）。由 PM 起草，Cap-Spec-3 阶段涉及契约变更时需创始人审批，然后"冻结"进入实现。
HPRDHuanlong PRD流程
唤龙风格的产品需求文档。在 Cap-Spec 冻结后由 PM + 设计共同完成，比传统 PRD 更强调"验收条件可机器校验"。
冻结Freeze / Lock标准
文档或契约通过评审后锁定，不再允许修改。要改必须走新 PR 重新评审。类似 Git 打 tag。
门禁Gate / Required Check标准
合并 PR 前必须通过的自动化或人工检查点。本站分四层：L1 格式、L2 确定性、L3 行为、L4 语义。详见 门禁层级详解。
BLOCKING / ADVISORYBlocking vs Advisory标准
BLOCKING = 不通过就不能合并；ADVISORY = 只提示不拦截。门禁每一层都明确标注类型。
pm-acceptancePM Acceptance Check流程
GitHub Required Status Check 之一，代表 PM 已在预览环境跑过验收场景。不是口头"我看过了"，是可审计的 CI 标志。
12 状态机12-State Workflow流程
能力包从立项到合并的 12 个显式状态（Proposed → Cap-Spec Draft → … → Merged）。任何时候问"这个包到哪一步了"都能准确回答。详见 12 状态详解。

—— 仓库与组件 ——

hl-contractsContracts Repo核心概念
契约仓库。全局规则、Schema、接口定义都在这里，由创始人守门。所有其他仓库依赖这里。
hl-platform / hl-factoryPlatform / Factory标准
平台后端（Kotlin/Spring）+ 能力包工厂（脚手架与生成器）。详见 仓库目录。
hl-console-native / hl-appConsole / Mobile App标准
前端分治（SAAC §4.1）：hl-console-native = 平台控制台 C-L1，SwiftUI + 光合设计系统（GHKit），原生 macOS / iOS 26；hl-app = 租户/消费者端 C-L2，Flutter + Dart + DTCG，覆盖 Desktop / Mobile / Web。
hl-design-system / 光合 GHKitDesign System标准
前端设计系统双线：hl-design-system 服务 C-L2（Flutter + DTCG Token 真源）；光合设计系统（GHKit） 服务 C-L1（SwiftUI 组件库，随 hl-console-native 一起演进）。两者各自为本栈的 Token / 组件 / 规范单一真相源。

—— 软件工程标准术语 ——

SSOTSingle Source of Truth标准
单一真相源。一条信息只有一个权威来源，别的地方要么引用要么生成——不允许存两份会漂移的副本。
AI-FirstAI-First Development核心概念
流程和工具链都按"AI 写代码"为默认场景来设计，而不是"人写代码 + AI 辅助"。契约先行、测试先行、机器可验证，就是为了让 AI 能稳定跑通。
镀金Gold-Plating反模式
工程师擅自添加"用户没要的、显得厉害的"功能。本组织明令禁止，因为会污染契约和验收场景。
哨兵Sentinel标准
CI 里的轻量检查脚本，定期跑一次确认"系统还活着 / 契约没漂移"。不是主测试，是"守夜人"。
Required Status CheckGitHub 必过检查标准
GitHub 合并保护规则：指定的 CI job 没显示绿勾，PR 就点不了 Merge。本站的四层门禁全部通过这个机制落地。
可观测性Observability标准
系统运行时能被看见——日志、指标、链路追踪三板斧。运维团队的核心输出之一。
蓝绿 / 金丝雀Blue-Green / Canary标准
发布策略。蓝绿 = 双环境切流；金丝雀 = 先给 1% 用户试，稳定后扩大。目标都是零停机 + 可快速回滚。

—— 组织与会议 ——

创始人Founder角色
本组织治理权与全局语义主权的最终持有人。不审全部代码，只审契约、架构、能力包范围。
PMProduct Manager角色
产品经理。负责能力包内的业务语义、验收场景、PM-Acceptance 签字。不写代码但会写 Cap-Spec。
