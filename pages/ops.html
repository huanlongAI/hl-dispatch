🚀 运维分工流程

⬤ 运维主导或参与   ○ 待命/观察

一、你在新模型中的角色定位

运维 — Release & Infra Owner

你是发布流水线和基础设施的 owner。在 AI 驱动的新模型中，你的定位是：

  • CI/CD pipeline 的日常运维和故障排查
  • 环境管理（dev / staging / prod）的守护者
  • 发布执行与回滚的最后一道防线
  • 基础设施即代码（IaC）的维护——AI 辅助生成，你审核和执行

你拥有的：

  • 发布流水线的运维权
  • 环境管理权（配置、部署、监控）
  • 发布执行与回滚权
  • 基础设施变更权

你不做的：

  • 不做代码审查
  • 不做业务需求定义
  • 不做技术方案设计
  • 不做测试验收

二、你的核心产出

产物 说明 存放/记录位置
环境配置与部署脚本 dev/staging/prod 环境的部署配置 各仓库 .github/workflows/ 或 IaC 仓库
发布记录（release record） 每次发布的版本、时间、变更内容、回滚预案 Release Notes / 发布平台
监控告警配置 服务健康检查、性能告警、错误率告警 监控平台
基础设施变更记录 数据库变更、服务器调整、网络配置 变更管理系统

三、你在能力包流中的角色

你在五步流中只出现在 Post-Merge 阶段：

Step 1-5: PM / 工程师 / QA 完成能力包交付 → PR Merged
    ↓
Post-Merge: 你登场
    ↓
1. staging 环境部署
  ├── 自动或手动触发部署流水线
  ├── 确认服务启动正常、健康检查通过
  └── 通知 QA 开始 staging 回归验证
    ↓
2. QA staging 回归验证
  ├── QA 执行回归测试
  ├── PASS → 你可以推 prod
  └── FAIL → 暂停，等工程师修复
    ↓
3. prod 环境部署
  ├── 执行发布流水线
  ├── 确认服务正常
  ├── 监控告警就位
  └── 回滚预案确认
    ↓
4. 发布完成 → 状态转为 Released

四、发布策略

4.1 发布节奏

发布类型 节奏 说明
常规发布 跟随 merge queue 或固定窗口 能力包完成、Bugfix P1-P3
紧急发布 随时 Bugfix P0 紧急修复
治理变更 立即生效 治理/内核变更不等发布窗口

4.2 环境保护

  • 创始人定义发布策略和环境保护规则（GitHub Environments + Deployment Protection Rules）
  • staging 部署需要 QA 回归通过
  • prod 部署需要 staging 回归 PASS + 监控告警就位

4.3 回滚策略

  • 每次 prod 部署前确认回滚预案
  • 发布后 30 分钟内为观察期——重点监控错误率、响应时间、核心业务流程
  • 发现异常立即回滚，不等排查

五、你和其他团队的协作

你和谁 场景 你做什么
工程师 PR merge 后 你执行部署流水线
QA staging 部署后 通知 QA 回归验证；QA PASS 后你推 prod
PM PM 验收通过后 PM-acceptance 和 qa-verdict 都 PASS 后 PR 才会 merge，你只在 merge 后执行
创始人 高风险发布 创始人可能额外审批；环境保护规则由创始人定义

关键时序对齐：

工程师 PR merge → 你部署 staging → QA 回归 → 你部署 prod
                                  ↑
                          这个步骤不能跳过

六、CI/CD 基础设施维护

你需要维护的 GitHub Actions workflows：

Workflow 作用 你的职责
pm-hprd-pass.yml PM 确认 HPRD 确保 workflow 正常运行
qa-verdict.yml QA 验收判定 确保 workflow 正常运行
pm-acceptance.yml PM 产品验收 确保 workflow 正常运行
pr-classification.yml PR 分类校验 确保 workflow 正常运行
g026-acceptance-manifest-check.yml 验收覆盖率检查 确保 workflow 正常运行
部署 workflows staging / prod 部署 你编写和维护

日常维护：

  • 监控 CI 队列，确保门禁不卡队
  • GitHub Actions runner 的容量和性能
  • 自动化脚本的版本更新
  • CI 缓存策略优化

七、基础设施管理

7.1 环境矩阵

环境 用途 访问权限 部署方式
dev 开发和联调 全团队 自动（push 触发）
staging 预发布验证 全团队 merge 后自动或手动
prod 线上环境 受限 手动 + 环境保护规则

7.2 数据库管理

  • 数据库变更脚本走 PR 流程
  • 变更脚本需要在 staging 先验证
  • prod 数据库变更需要回滚脚本
  • 备份策略确保数据安全

7.3 监控体系

监控项 告警阈值 通知对象
服务健康检查 连续 3 次失败 运维 + 工程师
错误率 > 1% 运维 + 工程师
响应时间 P99 > 2s 运维 + 工程师
CPU / 内存 > 80% 运维
磁盘空间 > 85% 运维

八、在 AI 驱动模型中的你

AI 驱动不只是写代码的人用 AI。你也可以：

  • IaC 生成：用 AI 辅助生成 Docker / Docker Compose / GitHub Actions 配置（Phase 0 不引入 Kubernetes 与 Terraform），你负责审核和执行
  • 监控规则：用 AI 辅助分析日志模式，生成告警规则
  • 故障排查：用 AI 辅助分析错误日志，定位问题
  • 部署脚本：用 AI 辅助编写部署自动化脚本

但记住：AI 的输出是草稿，你是裁决者。所有基础设施变更必须经过你的审核。

九、Bugfix P0 紧急流程中的你

当 P0 紧急 Bug 出现：

Bug 报告 → 工程师直接修复 → 最小 review → CI 全绿 → qa-verdict PASS
    ↓
你：紧急部署流程
  ├── 确认修复 PR 已 merge
  ├── staging 快速部署 + QA 验证（可压缩）
  ├── prod 紧急发布
  ├── 监控确认修复生效
  └── 通知团队

P0 的特殊规则：

  • staging 验证可以压缩（QA 只验证修复点，不跑完整回归）
  • prod 部署不等常规窗口
  • 事后补 Root Cause Analysis

十、常见问题

Q：merge queue 满了怎么办？ A：检查是否有 CI 门禁卡住的 PR。如果是 CI 基础设施问题（runner 不够、缓存失效），你负责解决。如果是 PR 本身的测试失败，通知工程师。

Q：staging 回归发现问题但不确定是不是新引入的？ A：和 QA 沟通确认。如果是历史遗留 Bug，记录但不阻塞本次发布。如果是本次变更引入的，阻塞并通知工程师修复。

Q：创始人定义的环境保护规则有哪些？ A：GitHub Environments + Deployment Protection Rules，具体规则在仓库设置中。通常包括：prod 部署需要特定 reviewer approve、staging 环境允许自动部署等。

Q：我需要参与 Code Review 吗？ A：不需要。你的职责是部署和基础设施，不是代码审查。但如果 PR 涉及 CI/CD 配置文件（.github/ 路径），你可能会被 CODEOWNERS 规则自动拉入 review——此时你审的是"部署配置是否正确"，不是"业务逻辑是否正确"。

Q：我只有 1 个人，够吗？ A：在 AI-first 模型下，大量运维工作可以自动化。你的核心价值在于：审核 AI/工程师产出的部署配置、监控异常、紧急回滚。日常部署尽量自动化，你把精力放在非常规情况的处理上。

🛠 技术选型与学习入口

运维 1 人支撑全组织，靠的是"配置即代码 + 全链路可观测 + AI 辅助排障"。工具链收敛到下表这一套，不接受新增运维工具（每多一个都会指数级增加维护负担）。

工具类型选型理由官方教程
Docker + Compose容器开发/预发一致性；本地 compose 起全栈Docker Get Started · Compose Docs
ECS 4C8G + Docker Compose生产编排（Phase 0）SAAC-HL-EP-001 v1.1 P0-4 LOCKED：Phase 0 单 VM + Compose，不预置 Kubernetes；待 Phase 1 按负载评估再定 K8s / K3sDocker Compose 文档 · Compose File 规范
GitHub ActionsCI/CD与门禁层级原生集成；Deployment Protection Rules 落地四权分离GitHub Actions 文档
Prometheus指标时序数据库事实标准；Spring Boot Actuator 原生导出Prometheus 入门
Grafana可视化多数据源仪表板；配置可 Git 管理（Provisioning）Grafana Getting Started
Loki日志轻量日志聚合；与 Grafana/Prometheus 同家族，运维心智负担低Loki Get Started
OpenTelemetry追踪厂商中立的可观测性标准；后端 Spring Boot 原生支持OTel Java 文档
Caddy反向代理自动 HTTPS；Caddyfile 极简，比 Nginx 配置成本低Caddy 文档
Claude / CodexAI 助手根因排查、YAML/HCL 生成、异常日志解读Claude 模型说明
学习路径建议（1 个月上岗）
第一周：Docker + Docker Compose 熟练；本地 compose 起全栈、理解 Compose Profiles / healthcheck / depends_on。
第二周：GitHub Actions；配一条从 PR 到 ECS staging 的全自动流水线（build / push / compose up）。
第三周：Prometheus + Grafana + Loki 三件套；配好一套黄金指标仪表板。
第四周：跟一次完整的发布 + 一次故障演练；学会用 Claude 快速定位根因。
（Kubernetes / K3s / Terraform 不在 Phase 0 范围内，留待 Phase 1 按负载重新评估。）
