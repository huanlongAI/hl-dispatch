🚀 运维分工流程

⬤ 运维主导或参与   ○ 待命/观察

一、你在新模型中的角色定位

运维 — Release & Infra Owner

你是发布流水线和基础设施的 owner。在 AI 驱动的新模型中，你的定位是：

  • CI/CD pipeline 的日常运维和故障排查
  • 环境管理（dev / staging / prod）的守护者
  • 发布执行与回滚的最后一道防线
  • 基础设施即代码（IaC）的维护——AI 辅助生成，你审核和执行

你拥有的：

  • 发布流水线的运维权
  • 环境管理权（配置、部署、监控）
  • 发布执行与回滚权
  • 基础设施变更权

你不做的：

  • 不做代码审查
  • 不做业务需求定义
  • 不做技术方案设计
  • 不做测试验收

二、你的核心产出

产物 说明 存放/记录位置
环境配置与部署脚本 dev/staging/prod 环境的部署配置 云效流水线 / 部署仓 / IaC 仓库；GitHub workflow 由 INV-0 维护
发布记录（release record） 每次发布的版本、时间、变更内容、回滚预案 Release Notes / 发布平台
监控告警配置 服务健康检查、性能告警、错误率告警 监控平台
基础设施变更记录 数据库变更、服务器调整、网络配置 变更管理系统

三、你在能力包流中的角色

你在五步流中只出现在 Post-Merge 阶段：

Step 1-5: PM / 工程师 / QA 完成能力包交付 → INV-0 GitHub Merge
    ↓
Post-Merge: 你登场
    ↓
1. staging 环境部署
  ├── 自动或手动触发部署流水线
  ├── 云效流水线执行 flyway migrate（DD-DB-MIGRATION-FLYWAY）
  ├── 确认服务启动正常、健康检查通过（ddl-auto=validate 双保险）
  └── 通知 QA 开始 staging 回归验证
    ↓
2. QA staging 回归验证
  ├── QA 执行回归测试
  ├── PASS → 你可以推 prod
  └── FAIL → 暂停，等工程师修复
    ↓
3. prod 环境部署
  ├── 执行发布流水线（云效 flyway migrate job → 应用部署 job）
  ├── 确认服务正常
  ├── 监控告警就位
  └── 回滚预案确认（Schema 不倒流原则：前向修复 V{n+1} 或 DB 备份恢复）
    ↓
4. 发布完成 → 状态转为 Released

四、发布策略

4.1 发布节奏

发布类型 节奏 说明
常规发布 跟随 GitHub 合并后的常规窗口或固定窗口 能力包完成、Bugfix P1-P3
紧急发布 随时 Bugfix P0 紧急修复
治理变更 立即生效 治理/内核变更不等发布窗口

4.2 环境保护

  • 创始人定义发布策略和环境保护规则（GitHub Environments + Deployment Protection Rules）
  • staging 部署需要 QA 回归通过
  • prod 部署需要 staging 回归 PASS + 监控告警就位
  • 生产数据库迁移必须作为独立发布阶段执行：validate/info → backup → migrate → deploy → verify

4.3 回滚策略

  • 每次 prod 部署前确认回滚预案
  • 发布后 30 分钟内为观察期——重点监控错误率、响应时间、核心业务流程
  • 发现异常立即回滚，不等排查

五、你和其他团队的协作

你和谁 场景 你做什么
工程师 PR merge 后 你执行部署流水线
QA staging 部署后 通知 QA 回归验证；QA PASS 后你推 prod
PM PM 验收通过后 PM-acceptance 和 qa-verdict 都 PASS 后 PR 才会 merge，你只在 merge 后执行
创始人 高风险发布 创始人可能额外审批；环境保护规则由创始人定义

关键时序对齐：

INV-0 GitHub Merge → 你部署 staging → QA 回归 → 你部署 prod
                                  ↑
                          这个步骤不能跳过

七、基础设施管理

7.1 环境矩阵

环境 用途 访问权限 部署方式 域名 / 入口
dev 开发和联调 全团队 可复用现有本地 K8S 底座，按 namespace 隔离；具体资源配额由运维落地 https://api.hldev.pro123.cn/hk（如运维实际域名不同，以 #45 回填为准）
staging 预发布验证 全团队 可复用非生产 K8S / 共享中间件底座；必须与 prod token、DB、配置隔离 https://api.hltest.pro123.cn/hk 或独立 staging 域名（由运维回填）
prod 线上环境 受限 阿里云 ECS + JAR/二进制 + supervisor；当前建议至少 2 台业务 ECS + 1 台中间件/运维预留 ECS API：https://api.huanlong.group/hk
控制台：console.huanlong.group

实施原则：现阶段以“能安全跑起来”为目标，不强制运维采用固定底座。已有资源可以先用；但必须回填实际部署形态、访问链路、DNS/TLS、安全组、端口暴露、备份恢复、回滚、监控告警和风险项。

7.2 数据库管理

  • 数据库技术选型锁定 PostgreSQL 18；生产优先采用阿里云 PostgreSQL 托管实例，初始建议 4c / 8g / 100G 起，最终容量按非生产部署、压测和预算复核
  • 非生产可复用共享 PostgreSQL 底座，通过 db_name / schema / 权限隔离环境
  • Flyway 10.x LTS 是 Phase 0 起 DB Schema 治理唯一方案，migration 由研发提交并经 PR / CI 验证
  • 生产禁止应用启动自动改库；Flyway 迁移从应用启动拆出，发布链路按 build/test → validate/info → backup → migrate → deploy → verify 执行
  • migrate 失败后不得继续部署应用，不在 pipeline 自动 repair；只能人工恢复或由研发提交前滚修复迁移后重发
  • 运维负责生产备份、恢复能力、迁移窗口、异常处置和恢复演练口径；研发负责 schema 设计、锁表风险、兼容性和前滚/回滚方案

7.3 监控体系

监控项 告警阈值 通知对象
服务健康检查 连续 3 次失败 运维 + 工程师
错误率 > 1% 运维 + 工程师
响应时间 P99 > 2s 运维 + 工程师
CPU / 内存 > 80% 运维
磁盘空间 > 85% 运维

可观测口径：Phase 0 优先使用阿里云可观测平台承接主机、数据库、证书、日志与告警。Prometheus / Loki / Grafana / OTEL 可以作为自建候选，但不阻塞当前部署；如自建，需要额外评估 ECS 与数据盘成本。

7.4 后置与提前锁定规则

可以后置 / 动态调整 必须提前锁定
最终容量、最终采购清单、生产压测后的扩容策略、可观测平台最终托管或自建、未来能力包拆分后的独立服务拓扑 生产 API 域名与 base URL、/hk/** 路径前缀、环境隔离、JWT issuer/audience 或签名密钥隔离、Flyway 发布链路、数据库版本与备份恢复边界

八、在 AI 驱动模型中的你

AI 驱动不只是写代码的人用 AI。你也可以：

  • IaC 生成：用 AI 辅助生成云效部署配置、supervisor 模板、OpenResty / nginx 反向代理配置、备份恢复脚本；GitHub Actions / 保护分支配置由 INV-0 维护
  • 监控规则：用 AI 辅助分析日志模式，生成告警规则
  • 故障排查：用 AI 辅助分析错误日志，定位问题
  • 部署脚本：用 AI 辅助编写部署自动化脚本

但记住：AI 的输出是草稿。你裁决必须依赖运维实际资源和安全落地的事项；不依赖运维的域名语义、API 契约、客户端策略、认证边界等，由 INV-0 裁决，避免运维承担非技术性决策。

九、Bugfix P0 紧急流程中的你

当 P0 紧急 Bug 出现：

Bug 报告 → 工程师直接修复 → 最小 review → CI 全绿 → qa-verdict PASS
    ↓
你：紧急部署流程
  ├── 确认修复 PR 已 merge
  ├── staging 快速部署 + QA 验证（可压缩）
  ├── prod 紧急发布
  ├── 监控确认修复生效
  └── 通知团队

P0 的特殊规则：

  • staging 验证可以压缩（QA 只验证修复点，不跑完整回归）
  • prod 部署不等常规窗口
  • 事后补 Root Cause Analysis

十、常见问题

Q：待合并 PR 堆积怎么办？ A：先看 GitHub PR 上是不是有 Required Status Checks 没绿，或者 INV-0 还没到合并窗口。如果是 CI 基础设施问题（runner 不够、缓存失效），你负责解决；如果是 PR 本身的测试失败，通知工程师。

Q：staging 回归发现问题但不确定是不是新引入的？ A：和 QA 沟通确认。如果是历史遗留 Bug，记录但不阻塞本次发布。如果是本次变更引入的，阻塞并通知工程师修复。

Q：创始人定义的环境保护规则有哪些？ A：GitHub Environments + Deployment Protection Rules，具体规则在仓库设置中并由 INV-0 维护。你只读取发布约束并在云效侧执行，不直接改 GitHub 环境保护规则。

Q：我需要参与 Code Review 吗？ A：不需要。你的职责是云效部署和基础设施，不是代码审查。GitHub 仓库 / .github/ / Required Checks 变更由 INV-0 维护；你只在云效部署配置或运行资源受影响时提供运维意见。

Q：我只有 1 个人，够吗？ A：在 AI-first 模型下，大量运维工作可以自动化。你的核心价值在于：审核 AI/工程师产出的部署配置、监控异常、紧急回滚。日常部署尽量自动化，你把精力放在非常规情况的处理上。
