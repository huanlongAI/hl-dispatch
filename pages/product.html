📘 产品团队指南

唤龙平台 AI 驱动协作模型
PM + UI 设计师

适用角色：PM（3 人）+ UI 设计师（2 人） 关联文档：全局工作流 · 前端与设计指南 治理来源：TEAM-COLLABORATION-SPEC v2.1 §2.2 / §2.5 / §5.2 · UI-DELIVERY-SPEC v1.0

一、你在新模型中的角色定位

PM — 能力包语义 Owner

你是能力包业务语义的 SSOT（Single Source of Truth）。简单说：你定义"做什么"，不定义"怎么做"。

你拥有的权力：

  • 能力包业务语义的最终解释权（在创始人的契约包络内）
  • 验收场景定义权
  • 产品验收权（pm-acceptance，是 GitHub Required Status Check，不是口头确认）
  • HPRD 审批权（审的是"工程师有没有理解对"，不是审技术方案）

你不再做的事：

  • 不审 design.md（技术方案太深，不如不审）
  • 不审代码（交工程师互审 + AI 审计 + CI 门禁）
  • 不催工程师"你理解了吗"（HPRD 机制替代了口头确认）
  • 不管 AI 编码细节（交工程师）

UI 设计师 — 视觉/交互规范 Owner

你归属 PM 能力环，定义"长什么样"。在新模型中你的产出升级为仓库内的机读文件（Design Packet），不再是"群里发个链接"。

你同时参与两条线：

  • 能力包线：为每个能力包产出 Design Packet
  • Foundation 线：参与设计系统的 Token 定义、组件规范

二、你的核心产出

2.1 PM 的三份 Cap-Spec

产物 你做什么 放在哪
Cap-Spec-1（能力包规格书） 定义目标、非目标、业务规则意图、能力边界 hl-contracts/prd/core/{能力包}/
Cap-Spec-2（验收场景集） Case-ID + 输入 / 预期 / 验收方式（你是 owner，QA 是 collaborator） 同上
Cap-Spec-3（业务码提案） 新增/变更 reason_code 的建议。⚠️ 如涉及 reason_code 变更，必须创始人先批 hl-contracts/reasoncodes/（合入后）

关键规则：你的 Cap-Spec 必须在创始人定义的契约框架内。如果需要变更契约（新增 reason_code、修改 API 规范），先走创始人审批的治理前置门。

2.2 设计师的 Design Packet

Design Packet 是前端的真输入，取代了"群里说设计好了"。它是一份 YAML 文件，存在仓库里，有版本、可追溯。

存放位置：hl-app/docs/design-packets/{能力包}/design-packet.yaml

必填字段（12 个）：

字段 作用
capability_id 对应哪个能力包
figma_refs 设计来源 URL 列表
screen_ids 屏幕/页面集合
state_matrix 每个页面要覆盖的状态
token_version 绑定哪一版 token
component_refs 引用哪些基础组件
breakpoints 响应式/尺寸规则
a11y_notes 无障碍约束
api_contract_ref 依赖的 API/mock 版本
foundation_version_ref 依赖哪一版设计系统
foundation_change_required 是否需要先改设计系统
frozen_at + owner 冻结时间 + 责任人

版本钉死：Design Packet 一旦冻结，前端就按这个版本实现。如果设计要改，必须解冻 → 修改 → 重新冻结，不能一边改一边让前端追。

三、你的工作流程（能力包流五步中的角色）

Step 1：Cap-Spec Freeze — 你是主角

PM 编写 Cap-Spec-1（规格书）+ Cap-Spec-2（验收场景集）
    ↓
UI 设计师 补充视觉/交互设计
    ↓
判断：Cap-Spec-3 是否涉及 reason_code 变更？
  ├── 是 → 创始人先批准（治理前置门）→ 批准后你才能宣布 freeze
  └── 否 → 你直接宣布 Cap-Spec freeze

要点：

  • Cap-Spec freeze 前确保 UI 设计师已完成视觉/交互设计
  • 不要 freeze 后再改需求——改了就要解冻，工程师的 HPRD 和 design.md 全部作废

Step 2：HPRD 审批 — 你审"理解对不对"

工程师读完你的 Cap-Spec 后，会产出一份 HPRD（理解确认件）。你的审批内容只有一个：

"工程师的业务语义理解是否忠于你的 Cap-Spec？"

你不审：

  • 技术方案（那是 design.md，工程师互审）
  • 代码实现细节
  • 数据库设计、接口内部结构

HPRD 的最小模板（工程师填写，你审批）：

字段 你审什么
User-visible Changes 用户会看到什么变化——和你的 Cap-Spec 一致吗？
Unchanged Behavior 不变的行为——工程师有没有漏掉不该改的？
Out of Scope 明确不做什么——和你的非目标一致吗？
Acceptance Walkthrough 验收步骤——对应你的 Case-ID 吗？

审批通过 → pm-hprd-pass（流程内检查，非组织级 Ruleset）Required Status Check = PASS → 工程师可以开始实现

流程规则：pm-hprd-pass 未通过前，工程师不得开始 Step 3 实现。

Step 5：PM 产品验收 — 你是终审

工程师完成实现后，你在可交互演示版本上执行产品验收：

  • 对照 Cap-Spec-2 的验收场景逐一走一遍
  • 确认用户可见行为符合预期
  • UI 设计师协同参与视觉/交互维度验收

验收通过 → 你在 PR 上 approve → pm-acceptance Required Status Check = PASS → 进入 merge queue

四、你和其他团队的协作边界

你和谁 你做什么 对方做什么 边界在哪
工程师 提供 Cap-Spec，审批 HPRD，产品验收 HPRD + design.md + 代码 你不审技术方案，对方不定义业务需求
QA 提供 Cap-Spec-2（验收场景集），QA 是 collaborator QA 编写 acceptance-manifest 和验收测试 QA 独立出具 qa-verdict，你不干预
设计师 你定义"做什么"，确认 Design Packet 的业务边界 设计师定义"长什么样"，产出 Design Packet 视觉/交互细节交设计师，你不画图
前端 你通过 pm-acceptance 验收最终产品 前端按 Design Packet 实现 UI 前端技术方案你不审
创始人 Cap-Spec-3 涉及契约变更时请创始人审批 创始人定义契约包络和规则 你在包络内自由定义能力包语义

五、常见问题

Q：Cap-Spec freeze 后发现需求变了怎么办？ A：正式解冻 Cap-Spec（状态回退到 Cap-Spec Frozen），通知工程师和 QA，已有的 HPRD 标记为 INVALIDATED。修改后重新 freeze。不能偷偷改——改了门禁追溯不到。

Q：我能直接 merge Cap-Spec 的 PR 吗？ A：可以。prd/core/ 路径的 CODEOWNERS 是 @hl-pm-team，PM 可以直接 merge Cap-Spec PR。创始人可选 review 但不 block。

Q：设计师的产出在哪里冻结？ A：在 hl-app/docs/design-packets/ 下提交 design-packet.yaml。这是机读文件，有 frozen_at 字段标记冻结时间。

Q：HPRD 和 Cap-Spec 冲突怎么办？ A：以 Cap-Spec 为准。工程师修正 HPRD，你重新审批。Cap-Spec 是上游 SSOT，HPRD 只是工程师的理解确认。

Q：产品验收发现问题，但 QA 已经通过了？ A：你有独立的 pm-acceptance 权力。QA 验收的是"功能是否按规格工作"，你验收的是"产品是否满足用户需求"。两者可能有差异——你打回就是打回。

六、你的能力包分配

PM 负责能力包 分级
PM-A 销售流程、客户资产、售后服务 T1, T1, T3
PM-B 支付结账、营销、商品中心 T1, T2, T2
PM-C 供应仓储、数据看板、硬件POS、流程引擎 T2, T2, T3, T3

T1 先行：PM-A 和 PM-B 在 T1 阶段压力最大，各负责 1-2 个 T1 能力包。T2/T3 在 T1 完成后滚动推进。

UI 设计师 2 人横向支持所有有 UI 界面的能力包，不按能力包固定分配。
