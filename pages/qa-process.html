测试团队流程与分工详解

理解 QA 团队的职责、工作流程和验收标准

第一部分：QA 团队结构

唤龙平台的 QA 团队共 2 人，各自独立但紧密协作：

QA 架构师
1 人 · 战略与规范制定者
  • 制定验收标准框架
  • 设计测试流程和体系
  • 维护测试基线（Golden Tests）
  • 审阅和优化 acceptance-manifest
  • 参与5步流各个阶段的质量把控
  • 培养和指导 QA 工程师
QA 工程师
1 人 · 执行与验证者
  • 执行验收测试
  • 运行回归测试
  • 维护自动化测试脚本
  • 出具 qa-verdict（质量判定）
  • 参与 Widgetbook 评审
  • 跟进测试缺陷和补救
👑 共同职责：质量守门人
独立质量判定 · 不受其他团队影响
  • qa-verdict 独立签署 — QA 出具的质量判定不受 PM 需求或工程师进度影响
  • 验收权独立 — 定义什么是"完成"，完全由 QA 判定
  • 缺陷判定权 — 是否为 Bug、是否打回、缺陷优先级由 QA 决定
  • 变更影响评估 — 评估代码变更对测试覆盖的影响
💡 核心原则

QA 是独立的质量守门人。在 5 步流中，QA 的 验收权 与 PM 的 需求定义权 分离，QA 和 PM 的验收判定独立进行且都是 Required Status Check。

第二部分：QA 工作流 - 5 步交付全景

QA 在能力包交付的 5 个步骤中的参与点和关键活动

1️⃣ Cap-Spec Freeze — 需求冻结

QA 的活动：

阅读 Cap-Spec，理解需求范围、验收范围、业务规则

关键输出：

验收范围清单、初步测试场景草案

❗ 常见遗漏：

QA 应该在这一步就开始思考验收测试的设计，但很多团队忽略了 QA 在此步的参与

2️⃣ Design + HPRD — 设计与需求详化

QA 的核心活动：

编写 acceptance-manifest.yaml（与 HPRD 同步进行）

关键产出：
  • acceptance-manifest.yaml — 所有测试场景定义
  • G-026 覆盖率检查 ≥ 60%（Pilot 阈值）
  • Widgetbook 审核（UI 组件库评审）
⏱️ 时间同步：

acceptance-manifest 应与 HPRD 并行编写，不能等 HPRD 完成后再写

3️⃣ Implementation — 实现阶段

QA 的活动：
  • 准备自动化验收测试脚本
  • 维护 Golden Test 基线（参考实现）
  • 可进行 Early Review（PR 非 blocking 阶段）
  • 与设计师、前后端同步进度
关键输出：

自动化测试脚本、Golden 测试基线数据

4️⃣ Gates + Acceptance — 门禁与验收

自动化门禁（L2/L3）：
  • L2：acceptance-manifest 覆盖率检查（G-026）
  • L3：自动运行验收测试，所有 status=covered 的场景必须 PASS
QA 人工验收：

出具 qa-verdict（Required Status Check），condition：所有 acceptance-manifest 中标记为 covered 的场景测试通过

✅ 关键规则：

qa-verdict = PASS 的充要条件是 acceptance-manifest 中所有 covered 场景的测试通过，与 PM 需求是否满意无关

5️⃣ PM Acceptance + Merge — PM 验收与合并

QA 的角色：

不直接参与此步（保持独立性：QA verdict ≠ PM acceptance）

但若 PM 打回：

PM 发现问题并打回时，QA 需要重新验收确认修复是否影响已测试的场景

ℹ️ 分离原则：

QA 的验收权和 PM 的需求验收权是独立的两道门禁，互不影响

第三部分：acceptance-manifest.yaml 规范

QA 定义验收场景的标准格式和各字段含义

capability_id: biz.flow.sales version: "1.0" scenarios: - id: SC-001 title: 正常下单流程 steps: [打开商品页, 点击购买, 确认支付] expected: 订单创建成功，状态为 PENDING test_class: integration status: covered # covered | deferred | not-applicable - id: SC-002 title: 库存不足场景 steps: [打开商品页, 商品已售罄, 点击购买] expected: 系统提示"库存不足"，订单创建失败 test_class: integration status: deferred # Pilot 后续迭代处理

字段说明

字段 含义 示例 规则
id 场景唯一标识 SC-001 格式：SC-{3位数字}
title 场景的人类可读标题 正常下单流程 简洁明了，体现核心行为
steps 操作步骤序列 [打开页面, 点击按钮, 确认] 按执行顺序列出
expected 期望结果 订单创建成功 必须可断言
test_class 测试分类 integration / unit / e2e 影响自动化框架选择
status 场景覆盖状态 covered | deferred | not-applicable 见下方详解

Status 字段详解

covered

此场景在本能力包中实现且需要自动化测试覆盖。CI 运行时必须 PASS，qa-verdict 的前提条件。

deferred

此场景在本能力包中实现，但由于复杂性、依赖问题等原因，暂不做自动化覆盖，后续迭代处理。QA 人工验证过即可。

not-applicable

该场景不适用于本能力包（例如：边界条件、特殊场景、前置条件不满足等），可以在 manifest 中标记但不参与覆盖率计算。

📊 覆盖率计算（G-026）

覆盖率 = covered 场景数 / (covered + deferred) 场景数。Pilot 阶段需要 ≥ 60%，后续版本可根据项目成熟度提升。

第四部分：QA 的代码所有权足迹

QA 在不同仓库中的代码审阅权和责任范围

hl-design-system 仓库
  • /widgetbook/** — UI 组件库的可视化文档和交互演示
  • /test/golden/** — Golden Test 基线数据
  • /test/a11y/** — 无障碍（Accessibility）测试
hl-app 仓库
  • /test/acceptance/** — 验收测试脚本
  • /test/golden/** — Golden Test 基线
hl-platform 仓库（后端）
  • /acceptance/** — API 验收测试
  • /tests/** — 集成测试
⚠️ CODEOWNERS 权限

所有 CODEOWNERS 定义的路径变更都需要 QA 代码审核（Required Status Check），以确保测试基础设施的完整性和可靠性。

第五部分：QA 决策矩阵

根据变更类型（change_class），QA 采取不同的行动

change_class 场景说明 QA 的行动 qa-verdict
capability 新增或修改功能，需求在 Cap-Spec 中 执行完整验收流程（5 步全参与），编写 acceptance-manifest，自动化测试 ✅ 必需
bugfix 修复 bug，不涉及需求变更 回归测试 + 相关 manifest 场景更新。如果 bug 影响已有场景，补充测试确认 ✅ 必需
tech-improvement 技术改进，性能优化，重构（不涉及 UI 变更） 代码审阅，可选的集成测试。除非涉及 UI 变更，否则不强制 qa-verdict ⚠️ 可选
governance 治理文档、流程变更 QA 可参与评审，无需验收测试 ❌ 不需

第六部分：流程完整性检查与常见遗漏

创始人关注的 QA 流程可能存在的缺口，以及团队应该确认的关键问题

❓
QA 是否在 Step 1（Cap-Spec Freeze）就开始参与？
✅ 应该这样做：

QA 应在 Cap-Spec 评审时参与，理解需求范围和验收边界。不能等需求完全冻结后再看。

检查点：QA 是否在 Cap-Spec PR 中有评论或 approval？

❓
acceptance-manifest 是否与 HPRD 同步编写（而非之后补充）？
✅ 应该这样做：

manifest 应该与 HPRD 并行编写，而不是在 HPRD 完成后再补。这样可以尽早发现验收标准的歧义。

检查点：manifest 和 HPRD 是否在同一个 commit 或相邻 commit 中提交？

❓
前端 UI 的验收测试谁写？如何分工？
✅ 应该这样做：

QA 编写 acceptance-manifest 定义场景，前端或 QA 共同编写自动化测试脚本（取决于团队能力）。Golden Test 由 QA 架构师维护基线。

检查点：/test/acceptance/ 目录中谁有提交权？QA 是否在 PR review 中？

❓
多端测试策略是什么？（iOS / Android / Web 的覆盖优先级）
⚠️ 当前状态：

多端测试策略尚未在 DD-TEST 中明确定义。建议定义：

  • 优先级（Pilot 阶段 Web 优先）
  • 覆盖率差异（Web: 100%, Mobile: 70% 等）
  • 跨端共享测试用例的策略
❓
性能测试（Performance Testing）归谁负责？
⚠️ 当前状态：PROPOSED

性能测试的所有权暂未明确：

  • 是否归 QA（作为验收的一部分）？
  • 还是由后端或运维负责基准测试？
  • 如何定义性能验收标准（响应时间、吞吐量等）？

建议：待 T1 能力包试运行后明确此分工。

❓
安全测试（Security Testing）谁负责？
✅ 当前分工：

自动化安全扫描由哨兵 CI 层执行（SAST, Dependency Check），创始人负责人工安全审阅。QA 的责任是验证安全修复是否影响已测试功能。

检查点：CI 中是否配置了自动化安全扫描？创始人是否在 PR review 中确认安全问题？

❓
QA 和前端的 Golden Test 协作模式是什么？
✅ 应该这样做：

QA 架构师定义 Golden Test 的基线和更新策略，前端实现 UI 变更时需更新相应的 snapshot。QA 审核 snapshot 变更是否合理。

检查点：/test/golden/ 目录的 PR 是否有 QA 的 approval？

❓
如何处理跨越多个能力包的测试场景？
⚠️ 当前状态：PROPOSED

跨能力包的集成验收测试（如：用户注册 + 下单流程）还未有明确的所有权定义。

建议方案：

  • 各能力包的单独验收仍按各自 manifest 进行
  • 跨包集成测试作为 E2E 测试，由 QA 架构师在 Pilot 后规划
  • 暂不强制覆盖率要求
