测试团队快速指南

本节给测试组长与测试成员用于当天开工对齐：只看锁定技术栈、必交付物、PR 门禁和 qa-verdict 边界。详细流程继续读本页后续章节与 测试流程与分工详解。

身份质量与验收责任方，不是功能代码实现者。
主产物acceptance-manifest.yaml + 验收测试 + Evidence Pack。
主门禁G-026 解析 YAML；G-023 回放验收场景。
放行权qa-verdict 是 Required Status Check。
重点锁定：工程师不能放行自己实现的代码；PM 不能替代 QA 做质量判决；P0 紧急修复也不能绕过 qa-verdict。

1. 技术选型怎么用

  • JUnit 5.10.2 是唯一主测试框架与推荐 runner。
  • Kotest Property 5.9.1 只做属性测试补充，在 JUnit 5 runner 内调用；不要新增 FunSpec / StringSpec 风格测试。
  • Testcontainers 1.20.4 起真实 PostgreSQL 18.x 做集成测试与 Flyway 全量迁移回放；不要用 H2 替代。
  • Playwright 当前是学习入口，不是 PR 绿线；前端 E2E 是否入绿线以独立裁决为准。

2. 每个能力包必须补齐什么

  • 在 hl-platform/app/biz-*/acceptance/ 放置 acceptance-manifest.yaml。
  • 每个 Cap-Spec-2 Case-ID 必须能在 manifest 中找到映射。
  • API_EXEC、AUDIT_EXEC、UI_AUTO 场景进入 PR 前必须变成 COVERED，并绑定 testClass / testMethod。
  • UI_MANUAL 不伪装成自动化覆盖，进入 PM/UAT 里程碑验收。

3. PR 工作流怎么走

  • Cap-Spec / HPRD 冻结后，QA 同步编写 manifest，不等工程师写完再补。
  • 工程师实现期间，QA 维护验收测试与回归基线。
  • PR 打开后，L1 / L2 / L3 先跑；G-026 失败直接回工程师修 manifest 或覆盖关系。
  • QA 先完成证据审阅和问题复核，再由测试组长给出 qa-verdict。

4. 什么情况下必须打回

  • manifest 缺 Case-ID、字段不完整、自动化场景仍是 PENDING。
  • Testcontainers / JUnit 验收测试失败，或者证据包缺失关键日志。
  • 工程师要求口头放行，绕过 Required Status Check。
  • PM 验收满意但质量证据不足。PM acceptance 与 qa-verdict 是两件事。

当天执行清单

  1. 确认当前能力包的 Cap-Spec-2 Case-ID 已冻结。
  2. 创建或更新 acceptance-manifest.yaml，把每个 Case-ID 标出 mode、gate、status、testClass、testMethod。
  3. 对非 UI_MANUAL 场景补齐 JUnit 5 验收测试；需要真实数据库时使用 Testcontainers。
  4. PR 前自查 G-026：manifest 存在、字段完整、可自动化场景均为 COVERED。
  5. PR 后核验 L1 / L2 / L3 结果，先审阅 Evidence Pack，再由测试组长签出 qa-verdict。
推导链证据：SAAC-HL-001 v1.1 → BRIDGE-DERIVATION v1 → DD-TEST v1.2 → TECH-STACK-SPEC v3.4 → TEAM-COLLABORATION-SPEC v2.1 → 本站 Guide Layer。若本节与契约真源冲突，以 hl-contracts 中 DD-TEST 与 TECH-STACK-SPEC 为准。

测试团队工作流：先审阅，再批准发布

测试团队的工作流不是“跑完测试就点通过”。正确顺序是：确认输入、设计验收、执行验证、审阅证据、测试组长批准。任何一步缺证据，都不能进入 qa-verdict PASS。

Step 0 · 输入审阅

确认范围可测

测试组长检查 Cap-Spec-2 / HPRD 是否冻结，Case-ID 是否完整，验收口径是否可被自动化或 UAT 明确承接。范围不清先打回 PM / Engineer，不进入测试设计。

Step 1 · 测试设计

建立 manifest

QA 依据 Case-ID 编写 acceptance-manifest.yaml，标明 mode、gate、status、testClass、testMethod。可自动化场景必须规划到 COVERED。

Step 2 · 用例互审

先审 manifest

QA 内部先审 manifest 和测试类映射，确认没有漏 Case、错 mode、空 testClass。审阅通过后，才进入 PR 门禁执行。

Step 3 · 门禁执行

跑 L1 / L2 / L3

CI 运行确定性门禁与验收场景回放。G-026 失败代表 manifest 不合格；G-023 失败代表验收行为未通过；都必须回工程师或 QA 修正。

Step 4 · 证据审阅

审 Evidence Pack

QA 核对测试报告、失败日志、回归范围、UI_MANUAL UAT 记录和风险说明。证据不完整时只能 Request Changes，不能口头放行。

Step 5 · 批准发布

签出 qa-verdict

只有所有 COVERED 自动化场景通过、证据包完整、遗留风险被明确接受后，测试组长才能批准 qa-verdict = PASS。

批准边界：qa-verdict 只代表质量放行，不代表 PM 产品验收。PM acceptance 仍由 PM 负责；发布动作仍按运维/流水线流程执行。QA 可以拒绝不完整证据，不能替 PM 接受需求偏差，也不能替运维确认生产发布。

测试团队首个能力包接入前启动任务书 v0.4

大辉子最终复核：PASS

本任务书用于首个真实能力包进入 QA 放行 / 合并门禁前的启动准备。角色模型已按 Founder 口径收敛：测试团队内部只有测试组长（李云峰，兼架构师）与测试成员，不再拆分多层测试角色。

角色与签字权

  • 测试组长：统筹任务执行，制定 manifest / G-026 / Evidence Pack / qa-verdict 标准，拥有最终质量签字权。
  • 测试成员：按组长分配执行样例 manifest、JUnit 5 验收测试、证据收集、reviewer 支撑与演练。
  • 后端、运维、PM 只作为支持方，不拥有 QA verdict 签字权。

首包绿线范围

  • 首包前只锁定 API_EXEC / AUDIT_EXEC、G-026、G-023、Evidence Pack、qa-verdict。
  • UI_AUTO 在 Phase 1 前只作为非阻断候选或 UAT 里程碑，除非已有稳定自动化并经批准。
  • Playwright、PIT、SonarQube、QA 指标仪表板均不是首包阻断项。

D0 Kickoff

测试组长组织 30 分钟启动会，带读 DD-TEST / TECH-STACK / qa-workflow，确认 QA 与 PM acceptance 边界。

D1 Manifest

定义 acceptance-manifest.yaml 模板，样例 Case 集合覆盖 API_EXEC / AUDIT_EXEC / UI_AUTO / UI_MANUAL。

D2 G-026

测试组长拥有 G-026 QA 基建 DRI 与最终验收权；后端只做脚本实现与技术可运行性确认。

D3 JUnit 5

测试成员创建 JUnit 5 参数化验收测试骨架，失败输出必须包含 Case-ID、输入、期望、实际结果。

D4 Evidence Pack

补齐 G-026 / G-023 / manifest hash / reviewer / approver / timestamp；Approver 必须是测试组长。

D5 Dry Run

样例 PR 跑 FAIL → 修复 → PASS；覆盖缺证据、G-026 FAIL、G-023 缺失、G-023 FAIL、hash mismatch 等失败。

首个真实能力包准入门槛

  • 样例 PR 可用等价模拟 dry run；但首个真实能力包进入 QA 放行 / 合并门禁前，真实 qa-verdict required status check 必须接入目标 repo / branch protection / CI，并能实际阻断合并。
  • G-026 最小校验必须可运行；manifest 必须包含 schema_version / manifest_version。
  • 至少 1 个 JUnit 5 API_EXEC 样例可执行，至少 1 个 AUDIT_EXEC 样例可执行或有明确等价脚本。
  • P0/P1 豁免不得由测试组长单独批准，必须由 Founder 或指定治理 Owner 批准。
评审闭环：v0.3 经大辉子主脑二次评审为 CONDITIONAL PASS；v0.4 已关闭 2 个 P1 与 3 个 P2，最终复核结论 PASS。本站发布版以 v0.4 为准。

你是测试团队 Codex 助手。根据测试组长分配，从以下任务中选择一个执行：

A. G-026：输出 manifest YAML 最小校验规则、合法/非法样例、PASS/FAIL 输出格式。测试组长拥有 QA 验收权，后端只做技术实现确认。
B. JUnit 5：为 API_EXEC / AUDIT_EXEC 设计参数化验收测试骨架，使用 @Tag("acceptance")，失败输出包含 Case-ID、输入、期望、实际结果。
C. Evidence Pack：设计 Evidence Pack 模板与 qa-verdict checklist。QA Reviewer 可由测试成员担任，QA Approver 必须为测试组长。

输出：执行产物、验收清单、风险、需要支持的事项。

Codex 提示词 4：Dry Run 复盘

你是测试组长，负责 QA Bootstrap D5：样例 PR dry run 与复盘。

必须验证以下失败：
1. manifest 缺 Case-ID
2. API_EXEC / AUDIT_EXEC 自动化场景 PENDING
3. Evidence Pack 缺 G-023 结果
4. Evidence Pack 缺 Reviewer / Approver / Review Timestamp
5. manifest hash 与 PR commit 不匹配
6. G-023 结果存在但为 FAIL

输出：FAIL → 修复 → PASS 时间线、每次 FAIL 原因、修复动作、真实首包 required status check 接入状态、是否允许首个真实能力包进入 QA 放行 / 合并门禁的建议。

Codex 提示词 5：云效流水线取证

你是测试团队 Codex 助手，负责在云效 / Codeup 消费仓库并整理 QA Bootstrap 取证。

请根据当前 Codeup 分支、云效流水线 run、GitHub PR / 任务书，输出流水线取证报告。

必须检查：
1. 当前 Codeup 分支与 commit SHA。
2. 对应 GitHub PR 或任务链接。
3. 云效流水线 run URL、触发 commit、执行时间。
4. L1 编译 / 类型 / 格式结果。
5. L2 契约 / 模块 / G-026 结果。
6. L3 单元 / 集成 / G-023 / Testcontainers 结果。
7. Evidence Pack 是否包含 G-026、G-023、manifest hash、Reviewer、Approver、Review Timestamp。
8. GitHub main、Codeup main、云效构建 commit 是否需要发布前一致性校验。

边界：
- 云效 post-merge 部署流水线不能替代 GitHub pre-merge required check。
- 样例 PR dry run 可模拟 qa-verdict；真实首包前必须真实接入 required status check。
- G-026 QA 基建验收权归测试组长，后端只做技术可运行性确认。

输出：流水线取证表、失败分流建议、是否允许进入 qa-verdict 审阅、需要 Founder / 运维 / 后端支持的事项。
日报格式：日期 / 负责人 / D0-D5 状态 / 云效取证 / 今日完成 / 今日阻塞 / 后端支持 / 运维支持 / PM 支持 / 风险 / 下一步。

四、acceptance-manifest.yaml 详解

这是你最重要的产物。它是一份 YAML 文件，把 PM 的验收场景（Cap-Spec-2）映射到可执行的测试类。

结构

# acceptance-manifest.yaml
manifest_version: "1.0"
capability_id: "biz.flow.sales"
cap_spec_version: "Cap-Spec-Biz.Flow.Sales.v1.0"

cases:
  - caseId: "CASE-001"
    name: "正常销售流程完成"
    mode: API_EXEC          # API_EXEC | AUDIT_EXEC | UI_AUTO | UI_MANUAL
    gate: G-023
    testClass: "SalesFlowAcceptanceTest"
    testMethod: "normalFlowCompletes"
    status: COVERED         # COVERED | PENDING | EXCLUDED

  - caseId: "CASE-002"
    name: "库存不足时拒绝下单"
    mode: AUDIT_EXEC
    gate: G-023
    testClass: "SalesFlowEdgeCaseTest"
    testMethod: "insufficientStockDeniedWithAudit"
    status: COVERED

  - caseId: "CASE-003"
    name: "跨租户隔离验证"
    mode: UI_MANUAL
    gate: null
    testClass: null
    testMethod: null
    status: UAT_MILESTONE

字段规则

字段 必填 说明
caseId 是 对应 Cap-Spec-2 的 Case-ID
mode 是 API_EXEC / AUDIT_EXEC / UI_AUTO / UI_MANUAL
status 是 COVERED / PENDING / EXCLUDED；UI_MANUAL 可标记为 UAT_MILESTONE
testClass / testMethod 自动化场景必填 对应的验收测试类和测试方法
gate 是 自动化场景归属 G-023；UI_MANUAL 为 null

G-026 门禁会检查什么

G-026 是 L2 确定性门禁，对你的 manifest 做以下检查：

  1. R1：每个 app/biz-* 目录必须有 acceptance-manifest.yaml
  2. R2：manifest 中的 caseId 必须能对应 Cap-Spec-2 的 Case-ID
  3. R3：mode != UI_MANUAL 的可自动化场景必须是 COVERED 且 testClass 非空
  4. R4：G-026 只解析 YAML，不解析 Kotlin 源码，不读取 JaCoCo / Codecov 覆盖率数字

UI_MANUAL 不进 PR 门禁：需要人类操作判断的 UI 场景进入 PM/UAT 里程碑验收，不伪装成自动化覆盖。

五、你在能力包流中的时间线

Step 1: PM freeze Cap-Spec
  └── 你收到 Cap-Spec-2（验收场景集）
      你是 collaborator/reviewer，可以提建议但 PM 是 owner
    ↓
Step 2: 你开始同步编写 acceptance-manifest.yaml
  ├── 把 Cap-Spec-2 的每个 Case 映射为 manifest 条目
  ├── 初始状态大部分为 pending
  └── 和工程师同步：了解技术方案，确认验收测试的切入点

  ├── JUnit 5 参数化测试（AcceptanceMode ≠ UI_MANUAL）
  ├── 基于 acceptance-manifest 逐个覆盖
  └── pending → covered（更新 manifest）
    ↓
Step 4: 门禁 + 验收（你是关键角色）
  ├── 云效自动跑 L1-L3，并回推 GitHub PR Status Check
  ├── QA 团队独立执行验收测试套件并提交 evidence
  ├── 验证通过 → 测试组长在 PR 上 approve → qa-verdict = PASS
  └── 验证失败 → 测试组长 Request Changes，写明失败原因
    ↓
Step 5: PM 产品验收（你的 verdict 已经 PASS，等 PM）
    ↓
Post-Merge（INV-0 GitHub Merge 后）: staging 环境回归验证
  └── 你确认回归测试全过 → 运维推 prod

六、qa-verdict 怎么出

qa-verdict 是 GitHub Required Status Check。测试成员负责验收执行和证据收集，测试组长负责最终签字或打回。

  1. 测试成员执行验收测试、回归测试并在 PR 中提交 Evidence Pack
  2. 测试组长复核证据后在 PR 的 Review 中选择 Approve
  3. GitHub PR 的检查服务识别测试组长 approve → qa-verdict = PASS
  4. 如果发现问题 → 测试组长选择 Request Changes → qa-verdict 保持 FAIL

出具标准：

  • acceptance-manifest 中所有 COVERED 自动化场景的测试全部 PASS
  • 没有 P0/P1 级别的缺陷遗留
  • 回归测试无新增失败
  • 如果是 UI 相关：golden/visual diff + a11y 也要通过

铁律：qa-verdict 是测试组长的独立质量判断。PM 不能要求"先过"，工程师不能催"赶紧通过"。质量不过关就不过关。

十二、常见问题

Q：PM 的 Cap-Spec-2 写得不够清楚，我怎么写 manifest？ A：你是 Cap-Spec-2 的 collaborator/reviewer。在 Step 1 阶段就参与 review，提出"这个场景的预期结果不明确"之类的反馈。PM 有义务在 freeze 前解决。

Q：工程师催我"赶紧过 qa-verdict"怎么办？ A：你的 verdict 是独立权力。如果测试没全过、有遗留 Bug、覆盖率不达标，不要妥协。这就是四权分离的意义。

Q：PENDING 的场景什么时候要变成 COVERED？ A：只要该场景属于 API_EXEC、AUDIT_EXEC 或 UI_AUTO，进入 PR 门禁前必须映射到可执行测试并标为 COVERED。做不到就会被 G-026 阻断；UI_MANUAL 场景转入 PM/UAT 里程碑验收。

Q：staging 回归验证发现问题怎么办？ A：通知工程师 + PM。如果是 P0，启动紧急修复流程（bugfix P0）。运维暂停 prod 推送直到修复合入。

Q：AI 审计（L4）和我的验收有什么区别？ A：AI 审计是 Advisory（建议性），看代码质量和安全。你的验收是 Required（必须），看功能是否按规格工作。两者互补，不互相替代。
