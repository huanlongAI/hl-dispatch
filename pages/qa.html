🧪 测试团队分工流程

⬤ QA 主导或参与   ○ 待命/观察

一、你在新模型中的角色定位

QA — 质量 Owner（Acceptance Owner）

你不再是"手动测试执行者"。你是验收自动化 owner + 质量判定者。

你拥有的独立权力：

  • 独立质量验证权——工程师不能放行自己的代码，你说不过就不过
  • 验收自动化权——acceptance-manifest 和验收测试套件是你的产物
  • 回归基线权——回归测试集的维护和更新
  • 发布质量 verdict——qa-verdict 是 GitHub Required Status Check，不是口头确认

铁律：执行（工程师）与审计（QA）分家。同一条变更的实现者和质量放行者不能是同一个人。

你不做的：

  • 不写功能代码
  • 不定义业务需求（Cap-Spec-2 的 owner 是 PM，你是 collaborator/reviewer）
  • 不设计技术方案

二、团队内部分工

角色 人数 职责
QA 架构师 1 acceptance-manifest 标准制定、测试基础设施搭建、验收框架设计、质量度量体系、CI 报告体系
QA 工程师 3 验收测试编写与执行、回归测试维护、测试证据收集、qa-verdict 出具

QA 团队横向覆盖所有能力包，不按能力包固定分配。

产物 谁做 作用 存放位置
acceptance-manifest.yaml QA 验收映射清单（每个能力包一份） hl-platform/app/biz-*/acceptance/
验收测试套件 QA 可执行的验收测试 同上
回归基线 QA 回归测试集 hl-platform/app/biz-*/acceptance/
Quality Verdict QA qa-verdict Required Status Check PR Status Check
测试证据 QA + 工程师 Evidence Pack（合并前验收包） PR 附件

四、acceptance-manifest.yaml 详解

这是你最重要的产物。它是一份 YAML 文件，把 PM 的验收场景（Cap-Spec-2）映射到可执行的测试类。

结构

# acceptance-manifest.yaml
capability: biz.flow.sales
version: "1.0"

scenarios:
  - id: CASE-001
    description: "正常销售流程完成"
    status: covered          # covered | pending | not_applicable
    test_class: "SalesFlowAcceptanceTest"
    cap_spec_ref: "CAP-SALES-001/Case-001"

  - id: CASE-002
    description: "库存不足时拒绝下单"
    status: covered
    test_class: "SalesFlowEdgeCaseTest"
    cap_spec_ref: "CAP-SALES-001/Case-002"

  - id: CASE-003
    description: "跨租户隔离验证"
    status: pending
    test_class: ""            # 待编写
    cap_spec_ref: "CAP-SALES-001/Case-003"

字段规则

字段 必填 说明
id 是 场景唯一标识
description 是 场景描述
status 是 covered（已覆盖）/ pending（待编写）/ not_applicable（不适用）
test_class covered 时必填 对应的测试类名
cap_spec_ref 是 对应 PM 的哪个 Cap-Spec 的哪个 Case

G-026 门禁会检查什么

G-026 是 L2 确定性门禁，对你的 manifest 做以下检查：

  1. R1：每个 app/biz-* 目录必须有 acceptance-manifest.yaml
  2. R2：status=covered 的场景必须有非空的 test_class
  3. R3：status 只允许 covered / pending / not_applicable
  4. R4：covered 占比必须达到 60%（Pilot 阈值，后续可能升至 80%）

not_applicable 不计入分母：如果一个场景确实不适用当前阶段，标 not_applicable 不影响覆盖率。

五、你在能力包流中的时间线

Step 1: PM freeze Cap-Spec
  └── 你收到 Cap-Spec-2（验收场景集）
      你是 collaborator/reviewer，可以提建议但 PM 是 owner
    ↓
Step 2: 你开始同步编写 acceptance-manifest.yaml
  ├── 把 Cap-Spec-2 的每个 Case 映射为 manifest 条目
  ├── 初始状态大部分为 pending
  └── 和工程师同步：了解技术方案，确认验收测试的切入点
    ↓
Step 3: 工程师实现时，你同步编写验收测试套件
  ├── JUnit 5 参数化测试（AcceptanceMode ≠ UI_MANUAL）
  ├── 基于 acceptance-manifest 逐个覆盖
  └── pending → covered（更新 manifest）
    ↓
Step 4: 门禁 + 验收（你是关键角色）
  ├── CI 自动跑 L1-L3（含 G-026 manifest 检查 + G-023 验收回放）
  ├── 你独立执行验收测试套件
  ├── 验证通过 → 在 PR 上 approve → qa-verdict = PASS
  └── 验证失败 → 写明失败原因 → 打回
    ↓
Step 5: PM 产品验收（你的 verdict 已经 PASS，等 PM）
    ↓
Post-Merge: staging 环境回归验证
  └── 你确认回归测试全过 → 运维推 prod

六、qa-verdict 怎么出

qa-verdict 是 GitHub Required Status Check。你的操作方式：

  1. 在 PR 的 Review 中选择 Approve
  2. GitHub Actions 自动检测到 QA 团队成员 approve → qa-verdict = PASS
  3. 如果你发现问题 → 选择 Request Changes → qa-verdict 保持 FAIL

出具标准：

  • acceptance-manifest 中所有 covered 场景的测试全部 PASS
  • 没有 P0/P1 级别的缺陷遗留
  • 回归测试无新增失败
  • 如果是 UI 相关：golden/visual diff + a11y 也要通过

铁律：qa-verdict 是你的独立判断。PM 不能要求你"先过"，工程师不能催你"赶紧通过"。质量不过关就不过关。

七、与工程师的测试分工

测试类型 谁写 谁跑 说明
单元测试 工程师 CI 自动 governedAction 五条必过 + domain 逻辑
集成测试 工程师 CI 自动（Testcontainers） 端到端
验收测试 你 CI 自动 + 你手动确认 基于 Cap-Spec-2 验收场景
回归测试 你 CI 自动 你维护回归基线
属性测试（kernel） 工程师 CI 自动 kernel 级别
属性测试（能力包边界） 你 CI 自动 能力包边界属性测试

边界清晰：工程师不写验收测试，你不写单元测试。各司其职。

  • /test/acceptance/** → @hl-qa-team（你独审）
  • /test/golden/** → @hl-frontend-engineers + @hl-qa-team（你共审）

十二、常见问题

Q：PM 的 Cap-Spec-2 写得不够清楚，我怎么写 manifest？ A：你是 Cap-Spec-2 的 collaborator/reviewer。在 Step 1 阶段就参与 review，提出"这个场景的预期结果不明确"之类的反馈。PM 有义务在 freeze 前解决。

Q：工程师催我"赶紧过 qa-verdict"怎么办？ A：你的 verdict 是独立权力。如果测试没全过、有遗留 Bug、覆盖率不达标，不要妥协。这就是四权分离的意义。

Q：pending 的场景什么时候要变成 covered？ A：在 Step 3 期间逐步覆盖。PR 提交时 G-026 检查覆盖率，Pilot 阶段阈值是 60%。100% 不现实，但核心场景必须 covered。

Q：staging 回归验证发现问题怎么办？ A：通知工程师 + PM。如果是 P0，启动紧急修复流程（bugfix P0）。运维暂停 prod 推送直到修复合入。

Q：AI 审计（L4）和我的验收有什么区别？ A：AI 审计是 Advisory（建议性），看代码质量和安全。你的验收是 Required（必须），看功能是否按规格工作。两者互补，不互相替代。

🛠 技术选型与学习入口

QA 的核心是可机器复算的 verdict——所有测试必须能在 CI 里跑出 green/red，不接受"我点过了没问题"。工具链按"单元 → 集成 → E2E → API 契约 → 覆盖率"五层配齐。

工具类型选型理由官方教程

JUnit 5兼容层与 Spring Boot Test 深度集成；用于需要标准 JUnit 的场景JUnit 5 User Guide
Testcontainers集成测试 + Flyway 全量回放用 Docker 起真实 PostgreSQL 18 跑测试；告别 H2 假数据库（Phase 0 不预置 Cache/外部 MQ）；L2 门禁下同时承担 Flyway 全量迁移回放（DD-DB-MIGRATION-FLYWAY），PR 阶段拦下格式错误与启动阻塞风险Testcontainers Guides
PlaywrightE2E跨浏览器 E2E 测试；Trace Viewer 可回放失败用例Playwright 官方入门
