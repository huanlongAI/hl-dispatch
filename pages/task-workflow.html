任务接收、回填与优先级指南

收到任务、催办、复审请求后，先判断真源、回填位置和授权边界
PM / 工程师 / Gate / QA / 运维

适用角色：PM（产品经理）、工程师、Gate（关口审查）、QA（质量保障）、运维。

核心结论：飞书只是提醒你去 GitHub。GitHub Issue（事项）、PR（Pull Request，拉取请求）和仓库文件才是任务、状态、证据和裁决的 SSOT（Single Source of Truth，唯一事实源）。

先记住这几条红线 Draft PR（草稿拉取请求）不等于 baseline（基线）。HPRD（Human-readable Product Design，人类可读产品设计）草稿不等于 pm-hprd-pass（PM HPRD 通过检查）。CI（Continuous Integration，持续集成）green、sentinel pass（哨兵检查通过）、云效 run（流水线运行）和飞书通知都不等于 Gate 通过。没有 Founder（创始人）或 Gate 明确 runtime（运行态）授权，不得创建实现工单或写代码。

一、这页解决什么问题

PM知道 Cap-Spec（Capability Specification，能力规格）该写到哪里，哪些只是 Contract Gap（契约缺口）候选。
工程师知道 HPRD 是理解确认件，design.md 才是技术方案，何时仍不能实现。
Gate知道证据和裁决必须回到 GitHub，不能用飞书或流水线绿灯替代。
QA知道证据、qa-verdict（质量判决）和阻塞项应落在对应 Issue / PR。
运维知道云效、发布和通知只是证据输入或投影，不是业务授权。

二、先找真源，不从飞书开工

你收到飞书提醒时

  • 先打开提醒里的 GitHub Issue / PR 链接。
  • 确认任务 owner（负责人）、范围、验收标准、禁止事项和回填位置。
  • 如果飞书文字和 GitHub 不一致，以 GitHub 为准，并在 GitHub 说明差异。

你收到催办时

  • 催办通常表示任务存在卡点，优先级高于普通同步。
  • 最低动作不是在飞书回复“收到”，而是在 GitHub 回填当前状态和预计时间。
  • 如果无法推进，直接写 BLOCKED（已阻塞）并列出需要谁裁决。

层级 真源 / 投影 你要做什么 不能误解成什么

飞书投影层 飞书消息、任务、群提醒 提醒相关人回 GitHub 看任务、补状态、补证据。 飞书已读、评论、完成不是真源，不授权 runtime。

四、标准回填模板

收到任务或催办后，优先在原 GitHub Issue / PR 评论区回填。状态从下面五种选一种，不要自造状态。

状态中文含义使用场景
READY已准备好已读完真源，知道下一步，可按任务要求推进。
PARTIAL部分完成已有进展和证据，但仍缺关键链路。
BLOCKED已阻塞缺权限、缺真源、缺裁决、缺证据或存在冲突。
READY_FOR_REVIEW可复审证据链已整理完，等待 Gate / Reviewer 复核。
DONE已完成任务要求全部满足，并附可验证证据。

五、不同角色怎么判断边界

PM 版

  • PM 负责 Cap-Spec 草稿、验收场景、Contract Gap 候选和待裁决问题。
  • PM 可以列字段、状态、原因码、通知和 OpenAPI（开放接口规范）缺口候选。
  • PM 不裁决正式契约，不把 Draft PR 写成 baseline，不授权工程开工。

工程师版

  • HPRD 是你对 Cap-Spec 的理解确认件，不是技术方案。
  • design.md 是技术方案，进入实现前仍要看 Gate / Founder 是否明确授权 runtime。
  • 无 runtime 授权时，只能做阅读、理解确认、gap review（缺口复核）和证据整理。

Gate / QA 版

  • Gate 结论和 qa-verdict 必须落在 GitHub，可引用仓库文件、PR checks 和证据链接。
  • AI 总结、飞书评论、下载目录、本地草稿不能替代 evidence（证据）。
  • 缺证据时写 PARTIAL 或 BLOCKED，不写 DONE。

运维版

  • 云效 run、发布日志、构建产物和环境状态是证据输入。
  • 通知工作流是 observer-only（仅观察投影）和 fail-open（失败不阻塞合并）。
  • 运维证据不能替代 PM 验收、Gate 裁决或 runtime 授权。

七、不要这样写

错误说法正确说法
飞书任务完成，所以 owner 已确认。owner 需要在 GitHub SSOT Issue 按模板回填，飞书完成只说明提醒投影完成。
Draft PR 已建，所以 baseline 已定。Draft PR 只是草稿入口，baseline 以 GitHub 合并状态和裁决记录为准。
HPRD 草稿看起来可以，工程可以开工。HPRD 草稿只说明理解确认在推进，runtime 授权必须另有 Founder / Gate 明确裁决。
CI green / 云效通过，所以 Gate 通过。CI 和云效是证据输入；Gate 通过需要 GitHub 上的明确 Gate 结论。
我先写代码，后面补授权。无 runtime 授权时不得创建实现工单或写代码，只能做前置理解、证据和缺口整理。
默认通知策略 页面更新、Issue 回填建议和 PR 反链可以先在 GitHub 完成。飞书默认不发送；只有 Founder 明确要求时，才做飞书投影，并在 GitHub 写回投影事件。
