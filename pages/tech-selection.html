2 · 决定性因素模型

选型评估走这六因子；任何技术决定必须在下列因子上可追溯：

① 业务阶段
Phase 0 最小可行（单 VM + 单库 + 单前端栈对齐），任何"向未来过度工程化"的选型一律剔除。
② 团队规模与能力
23 人（内部 21 + 2 PHP 补漏 + 创始人在外），后端 6 人 + 前端 5 人上限。不选择需要专职 SRE / 专职 DBA / 专职 Platform 团队才能驾驭的栈。
③ 契约驱动
Schema / API / 事件契约先行（Flyway · OpenAPI · Modulith Boundary）。选型必须支持"契约即门禁"。
④ AI-First 编码
默认 AI 生成初稿 + 工程师过 Gate。选型偏向有稳定文档、类型系统强、LLM 训练语料充分的主流栈。
⑤ 四环三面
四权分离 / 四环 / 12 状态 / L1-L4 门禁的落地约束——选型必须不破坏 Modulith 边界与门禁可验证性。
⑥ 版本锁与供应链
TECH-STACK-SPEC v3 锁死次要版本。优先 LTS / 主版本稳定 / 商业支持充分的组件。

10 · 迭代规则与变更日志

迭代规则

  1. 新增选型：必须先走 SAAC 推演（确认决定性因素映射到六因子）→ 提交创始人裁决 → 拿到 Ruling ID → 更新 TECH-STACK-SPEC v3 Lock-in Table → 回填本图谱。
  2. 版本升级：次要版本升级需架构师签署；大版本升级需创始人裁决（涉及 TECH-STACK-SPEC 版本号）。
  3. 状态变更（PENDING → LOCKED / DEFERRED → ACTIVE / ACTIVE → REJECTED）：必须留 Ruling ID 与日期，写入本页 §10.2 变更日志。
  4. 作废旧选型：禁止直接删除——标 DEPRECATED 并标注替代项与迁移截止日期。
  5. 冲突仲裁：站点与 TECH-STACK-SPEC 不一致时以 Spec 为准，站点内容走修正 PR；TECH-STACK-SPEC 与 SAAC 不一致时以 SAAC 为准。
